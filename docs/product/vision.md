# Vision — Claudesk

## Vision

**The problem.** Starting work on any given project takes minutes of repetitive setup: open a terminal, `cd` to the project, launch `claude`; open Sublime Text and load the project; open Sublime Merge and load the project again; sometimes open a second terminal and `cd` again. The cost is small in isolation but ruinous in aggregate — 20+ active projects rotate through the queue, 3–4 are in flight on any given day, and each context switch pays the setup tax again. The setup friction discourages quick-pop-in visits to older projects and breaks flow when juggling several at once. Compounding it: when several projects ARE in flight, finding the one waiting on input means clicking through windows or switching Spaces — a second-order tax on top of the launch tax.

**The solution.** Claudesk — a macOS-only, single-user, open-source "lite IDE" that puts everything in **one window with multiple virtual workspaces inside it**:

- **VSCode-style project picker** — pick from a list of known projects, click, and the full environment fires up in <10s. Picking a project opens it as a **workspace** inside the existing Claudesk window (a new tab/stage), not a new OS window.
- **One workspace = one project = one CC session.** A workspace contains: Claude Code (PTY-backed terminal) on the left, lite editor + git diff viewer on the right (Phase 3; placeholder in Phase 1), and the workspace's own header chrome (drive-mode selector, skill buttons). Single-window holds N workspaces concurrently.
- **Mission Control-inspired layout.**
    - **Center stage:** the currently-focused workspace, full-size, fully interactive (CC terminal, editor, etc.).
    - **Top filmstrip:** live thumbnails of every other open workspace, ordered. Each thumbnail shows the project name, an idle/running CC indicator, and a miniaturized rendering of the workspace contents. **Clicking a thumbnail promotes that workspace to center stage** and demotes the previous one to the filmstrip. The filmstrip can be **collapsed to a row of mini status tiles** (project name + idle/running dot, no live preview) to reclaim vertical space — toggle via window chrome.
    - The cross-window status indicator from the previous vision iteration is **replaced**: idle-vs-running of every workspace is now intrinsic to the filmstrip/tile row, always in view inside the Claudesk window itself.
- **Picture-in-picture mini player (user-toggled, display-only).** A small always-on-top secondary window the user can summon when the Claudesk window loses focus (different macOS Space, another app focused, Claudesk minimized). It shows the same idle/running status for every open workspace, so the "which project is awaiting input" question is answerable without bringing Claudesk forward. Display-only in v1 — clicking a tile does NOT bring the workspace forward; the user switches back to the Claudesk window normally (⌘Tab, Mission Control, Dock). Toggled on/off explicitly by the user from Claudesk; not auto-summoned.

  > **OPEN DESIGN QUESTION** (defer to `/product-research`): Even a user-toggled, display-only PiP doesn't fully solve "Claudesk is hidden and PiP is off." A small **menu-bar item** (always visible system-wide, native macOS surface, cheap to build) would. The trade-off is one more surface to maintain and a duplication of what the filmstrip/PiP already show. Research should determine: (a) is the menu-bar item warranted as a parallel always-on surface; (b) if yes, should PiP become click-to-focus (richer but more complex) and the menu-bar item stay display-only-with-click-to-bring-app-forward, or some other split.

- **Left half of each workspace: Claude Code**, in yolo mode by default, already `cd`'d into the project.
- **Right half of each workspace: a built-in lite editor + git diff viewer** covering the daily-use Sublime Text features (multi-cursor / column selection, Cmd+P fuzzy file finder, command palette for syntax selection, project-wide find/replace, split panes, minimap). The in-app editor is meant to **replace** Sublime Text for routine work, not coexist with it: a hotkey switches the right half between editor ↔ diff viewer (↔ second terminal). The Phase-1 "pop the real Sublime Text" hotkey is a **stopgap** that exists only until the in-app editor lands, then is removed. *(Sublime Merge gets no permanent pop either — the in-app diff viewer covers it; a `smerge` escape hatch could be added later only if the diff viewer proves insufficient.)*
- **Stateful CC controller, not just an embedded terminal.** Claudesk owns each workspace's CC process lifecycle, watches workflow state files (`workflow/.session.md`), and exposes workflow operations as clickable buttons.
- **A thin orchestration layer above CC.** Clickable buttons for every custom workflow skill — no typing slash commands. A single "Recycle Session" button performs the manual ritual (`/session-pause` → wait for completion → Ctrl+D → spawn fresh CC → `/session-resume`) in one click. Triggered on user judgment, never automatically.
- **Smart auto-resume on workspace open.** When a workspace opens (project picked from the picker), Claudesk inspects two source-of-truth signals — the presence of `workflow/.session.md` and whether CC has a resumable conversation for the project dir — and fires the right command without the user having to remember which one applies. Three branches:

  | State signal | Meaning | Fire |
  |---|---|---|
  | `workflow/.session.md` exists | Clean pause via `/session-pause` — explicit next-step recorded | `/session-resume` |
  | No `.session.md`, but CC has a resumable conversation for this project dir | Mid-step termination (closed Claudesk, powered off, crash) — CC remembers, workflow doesn't | `/resume` (CC native) |
  | No `.session.md`, no resumable CC conversation, OR last action was a terminal-close (ship/finalize/resolve with no follow-up queued) | Fresh start | `/session-start` |

  Edge cases: if BOTH signals fire (`.session.md` exists AND CC has a resumable conversation), prefer `/session-resume` — workflow context is richer than raw conversation history; the user can manually `/resume` afterward if needed. Terminal-close cases (feature-finalize, task-close, incident-resolve, product-finalize) naturally fall into the fresh-start branch because closing skills don't write `.session.md`. No staleness heuristic on `.session.md` — trust it; `/session-resume` itself can sanity-check. Per-workspace memory of "what to fire next" is intrinsic to the source-of-truth files; no separate "next-command" state is persisted.

- **Drive-mode selector + indicator in each workspace's header.** The four drive modes (1 step-by-step / 2 orchestrated / 3 autopilot / 4 full-autopilot) govern how aggressively the workflow orchestrator chains between skills. Today they're selected via prompt text with no visible reminder of the active mode. The center-stage workspace renders a small selector in its header — current mode always visible, click to change. Persisted per-project; mirrored into the active WIP file's `drive_mode:` frontmatter so Claudesk's UI and the workflow's pause-policy logic agree on a single source of truth.

Built with Tauri (Rust backend, web frontend) to stay genuinely lightweight. The single-window-with-many-workspaces model is also a friendlier fit for Tauri than N independent windows — one WKWebView, one Rust process, one PTY-host owning N child PTYs.

> 1. **Live thumbnail cost.** Each background workspace's terminal is still rendering. With 4+ workspaces open, the filmstrip alone could rival the cost of 4 separate windows we set out to consolidate. Research should compare: (a) throttled snapshots (`canvas.toDataURL` every 1–2s), (b) render-on-hover with a static last-frame between hovers, (c) some xterm.js-internal cheap render path. > 2. **PiP window mechanics on macOS via Tauri.** macOS does not give app windows true `AVPlayer`-style PiP — what we actually build is a small always-on-top secondary Tauri window that survives Spaces (`NSWindowCollectionBehaviorCanJoinAllSpaces`). Research should verify Tauri 2's window API exposes the needed `NSWindow` collection-behavior flags, document the path (Rust `tauri::window::WindowBuilder` + raw `NSWindow` calls via `objc2` if needed), and surface any code-signing / entitlement requirements.
> 3. **Menu-bar item.** See the question deferred above under PiP.
> 4. **Tab-shell patterns in Tauri.** Are there existing Tauri 2 patterns (community examples, plugin) for the multi-workspace-in-one-window shell, or do we build the tab/filmstrip chrome from scratch on top of the single Tauri window? Research should produce 1–2 prior-art references or confirm "build from scratch."

## Target Audience

**Single user: the author (Stayman).** This is a personal tool, designed against one person's exact workflow — heavy Claude Code use, the custom workflow-system skill set installed at `~/.claude/skills/`, daily use of Sublime Text, macOS-only, 20+ rotating projects. Open-sourced so anyone running a similar setup can use or adapt it, but no design concession is made for users who don't share the workflow.

## Success Metrics

1. **Time-to-productive on a project drops from ~minutes to <10s.** Click a project in the picker → a new workspace opens in the current Claudesk window with CC running in the project dir, editor loaded, ready to type.
2. **The clean-context ritual becomes one click instead of a 4-step manual dance.** "Recycle Session" replaces pause → Ctrl+D → fresh launch → resume.
3. **Never have to type a slash command again for common skills.** Every installed workflow skill is a clickable button in the right context.
4. **Finding the awaiting-input CC across multiple workspaces takes <1 second of visual scan, zero clicks, from inside the Claudesk window.** The filmstrip (or its collapsed mini-tile row) shows idle/running for every workspace at all times. Switching to that workspace is one click.
5. **Returning to a project is one click, regardless of how the prior session ended.** Whether the last session was paused with `/session-pause`, killed mid-step by a power-off, or cleanly closed at finalize, picking the project drops the user into the exact right resumption state without choosing between `/session-resume`, `/resume`, and `/session-start` manually. The active drive mode is always visible in the workspace header — no prompt-level ritual to recall it.
6. **Finding the awaiting-input CC when the Claudesk window is NOT in focus** (different macOS Space, another app foreground, Claudesk minimized) **takes <1 second of visual scan via the PiP mini player or its equivalent, zero clicks to assess.** Acceptance for this metric depends on the open menu-bar-item question — the research pass decides whether PiP alone meets it or a menu-bar item is also required.

## Core Principles

1. **Lite over featureful.** If it's not a daily friction, don't build it. Reject scope that turns this into a general-purpose IDE.
2. **CC-process-aware AND workflow-state-aware, not just terminal-embedding.** Claudesk understands each workspace's CC lifecycle (idle vs running) via Claude Code's official hook channel — `UserPromptSubmit` / `Stop` / `Notification` events written to `~/.claude/settings.json` — and parses workflow state via known files (`workflow/.session.md`, active WIP file frontmatter, `.claude/skills/`). It uses those signals to make the right decision on workspace open (which of `/session-resume`, `/resume`, or `/session-start` to fire), to keep its UI honest (drive-mode indicator reflects the WIP file's `drive_mode:` frontmatter, not a separate in-memory toggle), and to populate the filmstrip / PiP / (possibly) menu-bar status surfaces. It NEVER infers state by scraping the terminal's text output. This is the architectural line that separates this tool from "tmux with a nicer skin."
3. **The Sublime *features* are sacred — the external Sublime *window* is not.** The built-in editor must preserve the daily-use Sublime feature set (multi-cursor, Cmd+P, command palette for syntax, project find/replace, split panes, minimap) so that in-app editing is a genuine replacement, not a second-class compromise. The right half is the editing home; a hotkey switches it between editor ↔ diff viewer (↔ second terminal). Popping the *real* Sublime Text in a separate OS window is a **Phase-1 stopgap** (it bridged the gap before the in-app editor existed) and is **removed once the editor lands** — a separate Sublime window per tab would re-fragment the single-window workflow that the tab model exists to protect. *(Revised 2026-06-19; the prior wording treated the external pop as a permanent escape hatch.)*
4. **One window, many workspaces, fewer prompt-level rituals.** The user should never manage OS windows for Claudesk itself — multiple projects live as workspaces inside the single Claudesk window, switched via filmstrip thumbnails. Right-half swapping within a workspace (editor / diff viewer / second terminal) happens inside that workspace. Persistent controls live in the workspace header (drive-mode selector) or window chrome (filmstrip collapse toggle, PiP toggle). A state that affects how every skill runs should be visible at a glance and changeable with one click, not buried in prompt text the user has to remember to type.
5. **No script-per-project, no per-project config burden.** Adding or removing a project from the picker is trivial — no `.claudesk.json` per repo, no bespoke launcher per project.
6. **Status visibility scales beyond the foreground.** Whatever workspace the user is focused on, and whatever app or Space they're switched into, the idle/running state of every CC session should be reachable at-a-glance with at most one action (open the PiP, glance at the menu-bar). The cost of forgetting a project is going-cold context — the status surfaces exist to prevent that, and they must work both inside Claudesk and outside it.

## Anti-Goals

- Not a VSCode replacement or general-purpose IDE.
- Not cross-platform — macOS only.
- Not multi-user; no team or sharing features.
- Not a Claude Code fork — Claudesk wraps the official CLI.
- Not an extension ecosystem.
- **Not multi-window for Claudesk itself.** Multiple projects = multiple workspaces inside one window. (The PiP mini player and the menu-bar popover are the auxiliary surfaces; Claudesk itself stays single-window. The Phase-1 Sublime-popped window was a temporary stopgap, removed once the in-app editor lands — see Core Principle 3.)
- **Not PiP click-to-focus in v1.** Display-only. Click-to-focus is a Future Possibility if the display-only surface proves insufficient in dogfooding.

## Future Possibilities (Not in Initial Roadmap)

Noted here so the architecture leaves room for them; explicitly NOT scoped into the initial roadmap.

- **PiP click-to-focus.** Promote a workspace from a PiP tile click — bring the Claudesk window forward and switch center stage to that workspace in one action. Defer until display-only PiP has been used long enough to confirm the limitation is real.
- **Migrate workflow state-machine enforcement into Claudesk.** Today the state machine documented in `_ref/claude-customization/docs/product/transitions.md` (product / feature / task / incident workflows; SURFACE / ESCALATE / REDIRECT cross-level mechanisms; AUTO/PAUSE drive modes; back-loop guards) is enforced advisorily — purely through `~/.claude/CLAUDE.md`, the per-workflow `AGENTS.md` orchestrator files, and individual skill prompts. Tuned to acceptable consistency but not perfect. Claudesk, as a stateful CC controller that already watches `workflow/.session.md` and can read `.claude/skills/`, is well-positioned to host harder enforcement (e.g., warn or block on out-of-state skill invocations) and to render a GUI visualization of the current workflow state, valid next transitions, drive mode, pending SURFACE/ESCALATE items, and back-loop history. Build Claudesk so this is a natural future addition — don't paint into a corner that assumes CC is a black box.
- **Time tracking integration.** The existing `claude-time` hook system at `_ref/claude-customization/tools/claude-time/` logs Claude Code hook events to a local SQLite DB and produces per-project / per-session / per-day time reports (tool time, active time, reading/thinking/away buckets, cross-session typing-debit attribution). Claudesk can plug into this in two ways down the line: (1) **richer signals** — Claudesk-level events (workspace switched, panel swapped, Sublime popped, Recycle Session clicked, skill button clicked) feed the same SQLite DB to give a more precise picture than hook events alone; (2) **in-window visibility** — surface `claude-time report` output (today's totals, per-project breakdown) inside the Claudesk UI rather than requiring a separate terminal invocation. Particularly valuable for the multi-workspace use case where "where did this week actually go" is a real question. **Note:** the in-roadmap filmstrip + PiP status surfaces tap the same hook channel (`UserPromptSubmit` / `Stop` / `Notification`) that `claude-time` uses, so Claudesk will already be registering a hook handler in `~/.claude/settings.json` by the time this future integration matters — sharing or co-locating with `claude-time` should be straightforward.
