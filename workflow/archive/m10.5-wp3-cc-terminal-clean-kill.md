## Problem Statement

Closing a workspace or quitting Claudesk does **not** terminate its CC / shell session
promptly and cleanly, **and it silently loses the right-panel shell terminal's command history.**
There are **two coupled defects** in the same kill path:

**Defect A — slow / unclean kill (reproduced, latency).** `PtyCcSession::kill()` spends **~5+ seconds**
on every close where the session isn't sitting at an interactive prompt reading stdin, and the "SIGKILL
fallback" is not the immediate SIGKILL the code comments claim. A subagent/detached child could also
orphan (single-PID kill, never `killpg`), though that path did not deterministically reproduce.

**Defect B — LOST SHELL HISTORY (operator's primary concern; verified).** Closing the app without
typing `exit` in the right-panel terminal → that zsh session's command history is **not** written to
`~/.zsh_history`. **Root cause (empirically verified this spec):** the operator's zsh (system-default
`/etc/zshrc`, no framework) has `SHARE_HISTORY`/`INC_APPEND_HISTORY` **off** — zsh writes history
**only on a clean/hangup exit**, never incrementally. The current kill path terminates the shell in a
way that skips that save. Crucially, **which signal we use decides whether history survives** (verified
against a real `zsh -l -i` idle at the prompt):

| Termination | zsh saves history? |
|---|---|
| clean `exit` | ✅ SAVED |
| **SIGHUP** | ✅ **SAVED** (default hangup handler runs the history save; completes in **~20ms**) |
| SIGTERM | ❌ LOST |
| SIGINT | ❌ LOST |
| SIGKILL | ❌ LOST |

`killpg(pgid, SIGHUP)` to the shell's process group **also** saves (verified, ~20ms). So the fix for
Defect B is: **on kill, deliver SIGHUP (to the group) and give it a real grace window (≥~200ms, ≪ the
old 3s) for the save to land, before escalating to SIGKILL.** This makes Defect A's fix and Defect B's
fix the *same* change — a fast SIGHUP-first process-group kill — as long as we do **not** pick SIGTERM.

**Expected:** closing a workspace / quitting terminates the session promptly (bounded, short even when
busy), with no multi-second UI hang, no lingering `claude`/shell/subagent process, **and the shell
terminal's command history preserved just as if the user had typed `exit`.**

**Observed:** ~5s per-session kill latency on every non-idle close; the "force kill" is actually
SIGHUP→250ms→SIGKILL to a single PID; **and shell history is lost** because the tight 250ms window
after SIGHUP and the frequent fall-through to SIGKILL don't reliably let zsh's on-exit save complete.

## Reproduction Attempt (carried from feature-reproduce, F32)

**Surface:** minimal standalone Rust harness (`<scratchpad>/kill-repro/`) replicating the EXACT
`portable-pty` 0.9.0 spawn (`setsid()` group leader — `unix.rs:257`) + `PtyCcSession::kill()`
sequence. Backend process-lifecycle bug — the unit harness can't spawn a real busy `claude`; the
standalone harness is deterministic and faster for the mechanism.

**Outcome:** reproduced, **every-run**. `kill_elapsed_ms ≈ 5237–5277ms`; `reaped_via_exit = false`
in every scenario where the session isn't at an interactive read.

**Root-cause localization:**
1. **PRIMARY (deterministic) — the 3s `exit_command\r` grace is wasted on every non-interactive
   close.** A busy / mid-turn / not-reading-stdin session never consumes the `exit_command\r` write,
   so `try_wait` polls the full 3s to no effect and *always* falls through to the forced kill. This
   is the operator-observed hang. *(WBS hypothesis (2) — confirmed dominant.)*
2. **SECONDARY — the "SIGKILL fallback" is not an immediate SIGKILL.** `cc_session/mod.rs:512-514`
   comments say "force kill // SIGKILL", but `child.kill()` resolves to portable-pty's
   `ChildKiller for std::process::Child` (`lib.rs:340-372`): **SIGHUP first → 5×50ms=250ms grace →
   only then stdlib SIGKILL**. Adds ~250ms + a first signal CC could trap; the comment is misleading.
3. **NOT reproduced for a plain child; ARCHITECTURALLY POSSIBLE — orphaned subagent.** Both signals
   target the **single leader PID via `libc::kill(pid,…)`, never `killpg`**, though the child is a
   `setsid` PGID leader. A plain foreground `sleep` grandchild WAS reaped (parent death + PTY teardown);
   a `setsid`-detached case was inconclusive (harness artifact). Carry to live verify-human with a real
   subagent-spawning CC turn.
4. **PTY-reap corollary.** A surviving child keeping the slave PTY fd open prevents EOF → the
   `cc-exit-<id>` event (reader-thread EOF, `mod.rs:462`) never fires → a workspace that never signals
   "closed". Only bites if (3) fires on the real app.

Full artifact detail preserved in git history of this WIP; harness is throwaway (not committed).

## User Stories

- **As the operator, when I close the app without typing `exit` in the right-panel terminal, I want
  that shell's command history preserved into my next session — just like it would be if I'd typed
  `exit`.** *(Defect B — the primary reason this QoL item was raised.)*
- As the operator closing a workspace whose CC is mid-turn, I want the session to die **promptly**
  (no ~5s freeze) so closing feels instant.
- As the operator quitting Claudesk with several busy workspaces, I want the app to exit within a
  short bounded window, not stall ~5s.
- As the operator, I don't want a killed CC session to leave an orphaned `claude` (or subagent, or
  shell) process running in the background eating resources.
- As a maintainer, I want the kill code's comments to match what actually happens (no "SIGKILL" that
  is really SIGHUP-then-SIGKILL), so the next person doesn't reason from a false premise.

## Acceptance Criteria

The feature is done when:

1. **Shell history preserved (Defect B — primary).** Closing a workspace or quitting the app while a
   right-panel terminal has un-`exit`'d command history → those commands **are written to
   `~/.zsh_history`** (verified live: run commands in a Claudesk shell terminal, close the app WITHOUT
   typing `exit`, reopen a terminal / check `~/.zsh_history` → the commands are present). Equivalent to
   having typed `exit`. Achieved by delivering **SIGHUP** (not SIGTERM) with a grace window that lets
   the on-exit save complete (~20ms observed; window ≥~200ms).
2. **Bounded, short kill latency (Defect A).** Killing a **busy** (non-interactive) CC or shell session
   completes in well under the current ~5s — target **≤ ~1s** for the forced path (down from ~3s wasted
   grace + ~0.25s fallback). Killing an **idle** interactive session still gets a brief clean-exit
   chance (`/exit` / `exit`) and we don't regress that.
3. **No orphaned processes.** After killing a CC session that has spawned a child (subagent / tool /
   backgrounded process), **no `claude` or descendant process survives** — verified on the real `.app`
   at verify-human with `pgrep`. The kill reaches the **process group**, not just the leader PID.
4. **PTY reaped / exit event fires.** The `cc-exit-<id>` event fires after kill (reader thread hits
   EOF), so the frontend/registry sees the session as truly closed — no wedged "never-closed" workspace.
5. **Honest code.** The `kill()` comments + the `CcSession::kill()` doc + `arch.md`'s
   "sends SIGTERM… then SIGKILL" line describe the actual signal sequence used (which is SIGHUP-first,
   NOT SIGTERM — the SIGTERM in the current arch text is both wrong-as-built AND the wrong choice).
6. **`kill_all` / quit path unaffected in shape.** The parallelized `kill_all` (M4 WP2) and the
   FRONTEND-DECIDES quit path (M10.5 WP2: `CloseRequested`→`quit-requested`→`quit_now`→
   `perform_quit_teardown`) keep working; total app-quit latency drops with the per-session latency.
7. **Regression coverage.** A test at the reproducible seam that would have caught the reproduced cases
   (e.g. a `kill()`-signal/timing assertion; a shell-history-preservation check where the harness
   allows; the live process-reaping + history halves carried to verify-human per the backend-lifecycle
   verify-self convention).

## Out of Scope

- **Job-control / foreground-process detection on the raw shell terminal** — WP2 already scoped
  terminal "active" to CC-status-only for v1; WP3 does not add foreground-process introspection either.
  The kill just needs to reliably terminate whatever is in the session's process group.
- **Forcing `INC_APPEND_HISTORY` / rewriting the user's shell config** — REJECTED. Claudesk must not
  mutate the user's `~/.zshrc` or inject history options; SIGHUP-first is the non-invasive fix that
  works regardless of the user's config (and respects a user who deliberately disabled history).
- **A "save history on every command via the hook/PTY" mechanism** — out of scope; SIGHUP-at-close is
  sufficient and far simpler than tapping every command.
- **Changing the clean-exit *command* semantics** (`/exit` for CC vs `exit` for shell) — that WP9 P1.5
  distinction stays where a clean exit is actually achievable (idle interactive prompt).
- **A Recycle-Session / soft-restart path** — that's a future `CcSession::recycle()` (Phase 2 forward-look),
  not this bug fix.
- **Windows** — Claudesk is macOS-only; the fix uses POSIX process-group signals.

## Technical Constraints

- **No 3rd-party dependency** (§2 probe check: N/A — this is an internal process-lifecycle fix; no
  external API/SDK/service). No probe WP required.
- **`portable-pty` 0.9.0 facts (verified this spec):**
  - The child is spawned as a **`setsid()` session/process-group leader** (`unix.rs:257`) → its **PID
    equals its PGID**, and its children share that PGID (unless they `setsid` away).
  - `Child::process_id() -> Option<u32>` (`lib.rs:141`) exposes the leader PID → usable as the PGID for
    `killpg(-pgid, …)`.
  - `Child::kill()` = SIGHUP→250ms→SIGKILL, **single PID only**. There is **no** group-kill in the
    library; we must call `killpg` ourselves.
- **`libc` (or `nix`) is not yet a *direct* dependency** of `src-tauri` (portable-pty pulls `libc`
  transitively). A `killpg` fix adds `libc` as a direct dependency — a small, standard, arch-relevant
  addition (single new crate, no new webview/module/data store). *(Prefer `libc` — minimal, already in
  the tree transitively — over `nix` unless the plan finds `nix` ergonomics worth it.)*
- **The M3 CC status (`Running`/`AwaitingInput`) lives frontend-side** (`useWorkspaceStatus`), not in
  the backend `kill()` path. A "skip the grace when the session is known-busy" optimization therefore
  either (a) plumbs a busy-hint into the backend kill command, or (b) is unnecessary if we simply
  shorten/replace the blind grace with a fast signal-first sequence. **Lean (b): a signal-first
  sequence removes the need to know busy-ness at all** — see Design Direction.
- **zsh history is on-exit-only for the operator (verified).** System-default `/etc/zshrc`;
  `SHARE_HISTORY`/`INC_APPEND_HISTORY` off, `APPEND_HISTORY` on (append-at-write, not incremental).
  So Claudesk cannot rely on incremental saves — it MUST give the shell a clean-enough exit (clean
  `exit`, or **SIGHUP** + grace) for the save to run. **SIGHUP saves; SIGTERM/SIGINT/SIGKILL do not**
  (verified against `zsh -l -i` at prompt; save ~20ms). `killpg(pgid, SIGHUP)` also saves.
- **The fix must NOT depend on the friend-user's shell config.** A friend with `INC_APPEND_HISTORY`
  on already never loses history; a friend on bash saves on `SIGHUP` too (bash writes history on exit /
  `HUP`). SIGHUP-first is the config-independent correct behavior for the whole class (it's literally
  the "your terminal closed" signal every login shell is built to handle). No per-shell special-casing.
- **PiP/menu-bar/main-thread rule:** the kill path runs on `#[command]` / event threads and background
  `kill_all` threads — none touch AppKit window ops, so the main-thread-marshaling rule does not apply
  here. (Noted so the plan doesn't over-apply it.)
- **No design-prior fires.** All recorded priors are product-design/UX leans; this is a
  technical/lifecycle tradeoff (arch.md territory, explicitly excluded from design-priors). The WBS's
  `explicit-selectable-mode-over-inferred-mode` note was for **WP2** (active-detection), not WP3.

## Design Direction (for plan)

A **SIGHUP-first, process-group kill** fixes BOTH defects with one change. The Defect-B finding
(SIGHUP saves zsh history, SIGTERM does not) **decides the signal** — this is no longer a fork.

1. **Keep a short clean-exit attempt.** Keep the `exit_command\r` write + a *short* poll window
   (≤~500ms, down from 3s) — an idle interactive session reaps cleanly in that window (fastest path,
   also saves history); a busy one doesn't and we move on fast instead of burning 3s.
2. **Signal the process GROUP with SIGHUP.** On the forced path, `killpg(pgid, SIGHUP)`
   (pgid = `child.process_id()`, which == PGID because portable-pty `setsid`'s the child). SIGHUP:
   (a) saves zsh history (Defect B, ~20ms), and (b) reaches subagents/children in the group (AC-3) —
   the current single-PID kill can't do either.
3. **Grace, then SIGKILL the group.** Give SIGHUP a grace window (≥~200ms — the ~20ms save + margin;
   plan picks the exact value, small) then `killpg(pgid, SIGKILL)` for anything that ignored SIGHUP.
4. **Reap + confirm EOF.** After the group kill, `try_wait()` the leader to reap the zombie so
   `cc-exit-<id>` fires (AC-4). The master-PTY-drop-on-Session-drop path already assists this.
5. **Fix the comments + `arch.md`** to state the real SIGHUP-first sequence (AC-5). The current
   arch line ("sends SIGTERM… then SIGKILL") is both not-as-built AND the wrong choice for history.
6. Keep `kill_all`'s parallelism and the WP2 quit wiring unchanged in shape (AC-6); total latency drops
   because each per-session window shrinks from ~5s to ~<1s.

**Remaining forks the plan resolves** (none block planning — clear leans):
- Exact grace-window value after SIGHUP (≥200ms; lean ~300ms for margin) and the short clean-exit
  poll value (≤500ms; lean ~300–500ms).
- Whether to drop the pre-exit `exit_command\r` for **CC** specifically (CC has no shell history to
  save; SIGHUP alone terminates it). Lean: keep it uniform (cheap, and a clean `/exit` is the nicest
  CC teardown) — but the plan may simplify.
- `libc` vs `nix` for `killpg`/`kill`. Lean: `libc` (already transitive, minimal).
- **CC-under-SIGHUP behavior** is the one thing not empirically pinned in-spec (spawning a real
  `claude` to signal-test is heavy) — SIGHUP's default disposition is termination and the SIGKILL
  escalation backstops any trap, so risk is low; **confirmed live at verify-human** (kill a real CC
  turn, confirm prompt exit + no orphan).

## Plan resolution of the remaining forks (decided)

- **Clean-exit poll window:** **500ms** (down from 3000ms). An idle interactive session reaps in this
  window (fastest, cleanest, saves history via the shell's own `exit`); a busy one falls through fast.
- **Post-SIGHUP grace before SIGKILL:** **300ms** (~20ms observed save + 15× margin). Small enough that
  even the forced path is ≤~800ms total (500 + 300), well under AC-2's ≤~1s.
- **Keep the pre-exit `exit_command\r` uniform** for both CC (`/exit`) and shell (`exit`) — cheap, and
  the cleanest teardown when the session IS idle. The signal path backstops the busy case.
- **`libc`** (approved) as a direct dep for `killpg`/`kill` — already transitive via portable-pty.
- **Signal target:** the **process group** `killpg(pgid, …)` with `pgid = child.process_id()` (== PGID,
  portable-pty `setsid`). Fallback: if `process_id()` is `None` (shouldn't happen post-spawn), signal
  the single PID (best-effort — preserves today's behavior rather than no-op).

## Retrospect
- **What changed in our understanding:** The WP was framed as a single "unclean exit" bug, but reproduce-first + a mid-spec operator note revealed **two coupled defects** — the ~5s kill latency/orphan risk AND (the operator's *primary* concern) **lost zsh shell history on close-without-`exit`**. The decisive discovery was that **the signal choice determines whether history survives**: SIGHUP saves it, SIGTERM/SIGKILL don't — which inverted the spec's initial `SIGTERM`-then-SIGKILL plan and made one change fix both defects.
- **Assumptions that held:** the child is a `setsid` process-group leader (portable-pty), so `pgid == child.process_id()` and `killpg` reaches subagents; `libc` was a clean minimal dep; the reify-the-sequence-as-a-walkable-value seam made the policy unit-testable.
- **Assumptions that were wrong:** (1) the spec's `SIGTERM`-first lean — SIGTERM loses history; corrected to SIGHUP before build. (2) The "SIGKILL fallback" was never an immediate SIGKILL — the library's `child.kill()` is SIGHUP→250ms→SIGKILL to a *single* PID (the comments lied). (3) At verify-codify, the integration test's `HISTFILE`-env isolation silently failed because **`/etc/zshrc` unconditionally overrides `HISTFILE`** — had to isolate via `HOME` (root-caused via `/debug-minimal-harness`; captured as a memory).
- **Approach delta:** Implementation matched the plan (single phase, SIGHUP-first process-group kill, 500ms/300ms timings). The only deltas were in verify-codify: the committed shell-history test took 3 iterations to get right (`print -s`→executed `echo`→`HISTFILE`→`HOME` isolation), each a test-harness artifact, not a code change — the shipped `kill()` was correct from build and verify-human-approved.
