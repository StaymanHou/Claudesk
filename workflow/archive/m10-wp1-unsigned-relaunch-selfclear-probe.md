## Problem Statement

Milestone 10's central risk (research Risk 1, arch "THE central risk"): Claudesk is **unsigned/un-notarized** by locked decision, so when the in-app updater downloads `Claudesk.app.tar.gz`, extracts it, replaces the running bundle in place, and relaunches, macOS Gatekeeper may block the freshly-written unsigned bundle as **"damaged / cannot be opened"** unless its `com.apple.quarantine` xattr is cleared. This probe must **empirically settle**, on a REAL unsigned installed `.app`, whether the as-designed **self-clear** mechanism ‚Äî the app running `xattr -dr com.apple.quarantine <own-bundle-path>` on itself after `install()` but before `relaunch()` ‚Äî makes the relaunch open **clean**. It also must settle three sub-unknowns the flow depends on: (a) does `tauri-plugin-updater`'s `install()` relaunch synchronously (leaving no window for a pre-relaunch self-clear hook), or is there a seam to inject the xattr step before relaunch? (b) the **App-Translocation** behavior for a properly-installed `/Applications` bundle (translocation would break BOTH self-clear and the WP3 brew-path detection); (c) does the whole `createUpdaterArtifacts` ‚Üí minisign-sign ‚Üí `check()`/`download()`/`install()` path even function on an unsigned build. **Deliverable = a documented GO / FALLBACK verdict** written to `docs/product/wbs.md` "Probe outcomes", which decides whether M10 ships a seamless self-clear UX (GO) or the instruct-the-user quarantine dialog (FALLBACK), and reshapes WP2's install‚Üírelaunch step + WP4's dialog accordingly.

**Operator dependency (structural ‚Äî read before building):** the success criterion ("does the self-updated unsigned `.app` open clean past Gatekeeper?") is ONLY observable on a real `/Applications`-installed, Finder/Dock-launched, **non-translocated** unsigned bundle. `pnpm tauri:dev` and the MCP bridge inherit the launching terminal's trust context and **never reproduce Gatekeeper** (installed-build smoke-test convention, with force). The agent CAN do all of: minisign keypair generation, `createUpdaterArtifacts` build config, building two unsigned versions (a "from" + a "to"), hand-authoring `latest.json`, hosting the manifest+artifact, wiring the check‚Üídownload‚Üíinstall‚Üíself-clear‚Üírelaunch flow, and confirming minisign verify/reject. The **"did it relaunch clean?"** observation is the operator's ‚Äî it lives in Phase 3's verify-human and cannot be auto-skipped.

**Type=probe verify posture:** this is a spike, not a shipped user feature. The "code" produced (keypair, build config, a temporary updater-flow wiring, `latest.json`) is throwaway/foundational evidence, NOT the final M10 implementation (WP2 builds that fresh, folding in this verdict). verify-codify writes NO regression tests for probe scaffolding ‚Äî it records the verdict + working sequence as the durable artifact (WBS "Probe outcomes"). The private minisign key is stored locally and NEVER committed.
