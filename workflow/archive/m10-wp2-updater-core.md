## Problem Statement

Claudesk needs an in-app update *engine*: from inside the running app, check a remote manifest for a newer version, download the signed update artifact, verify it (minisign), install it in place, clear the unsigned bundle's quarantine xattr, and relaunch into the new version. WP1 (the probe) proved the mechanism empirically — most importantly the **macOS seam**: `tauri-plugin-updater`'s `install()` returns control *without* relaunching on macOS (unlike Windows), so there is a clean point to run the self-`xattr`-clear after `install()` and before `relaunch()`. WP1 also built the durable pure-core (`resolve_bundle_path` / `quarantine_clear_command` / `clear_own_quarantine`, 6 unit tests) and confirmed minisign verify/reject.

WP2 turns that proven mechanism into the **production flow** — no UX polish, no brew gate, no persistence prefs (those are WP3/WP4). It replaces the throwaway WP1 probe scaffolding (`updater_probe` module + `UpdaterProbePanel`) with a clean production `updater` module + a **minimal temporary "Check & update" trigger** (operator decision 2026-07-17), so the whole path is smoke-testable at WP6 before WP4's polished control surface exists.

## User Stories

- As the operator (dogfooding a direct-download install), I want to click one control that checks for a newer Claudesk, and — after I confirm — downloads, verifies, installs, and relaunches into it, so I don't have to manually re-download and re-`xattr` on every release.
- As the operator, I want a tampered or wrong-key update artifact to be **rejected** (never installed), so a corrupted or spoofed download can't replace my running app.
- As a future WP (WP3/WP4/WP5), I want the update flow to live behind a **stable seam** (production `updater` module + typed commands) that the brew-gate can short-circuit, the polished UX can wrap, and `/release` can publish artifacts matching the config, so those WPs plug in rather than reshape.

## Acceptance Criteria

The feature is done when:

1. **Deps + config are production-shaped (not probe-scaffolding).** `tauri-plugin-updater` + `tauri-plugin-process` remain registered (already in place from WP1); `tauri.conf.json` has `bundle.createUpdaterArtifacts: true`, `plugins.updater.pubkey` (the real WP1 pubkey), and `plugins.updater.endpoints` → the static GH-Releases `latest.json` URL (`https://github.com/StaymanHou/Claudesk/releases/latest/download/latest.json`). `capabilities/default.json` (+ `tauri.dev.json` inline cap) grant `updater:default` + `process:default`. Committed version stays `0.2.5`.

2. **A production `updater` Rust module exists** (`src-tauri/src/updater/`, mirroring the pure-core/IPC-shell split used across the codebase), **absorbing WP1's durable self-clear core** (`resolve_bundle_path` / `quarantine_clear_command` / `clear_own_quarantine` + their 6 tests move in from `updater_probe`), and exposing typed Tauri commands for the flow.

3. **The full flow is wired and correct:** a frontend caller (or the backend command) runs `check()` → (if update available) `download()` (progress surfaced) → `install()` → **self-clear quarantine on the own bundle** (the WP1-proven insertion point: after `install()` returns Ok, before relaunch) → `relaunch()`. Uses the **split `download()` then `install()`** (not `downloadAndInstall`) so a future cancel has a clean boundary (cancel = don't call `install()`; the running app is untouched until install).

4. **minisign verification is live and rejects bad artifacts.** Pointing the flow at an artifact whose `.sig` is tampered/wrong-key makes the download/verify step FAIL and the flow does NOT proceed to install — proven by a test (or a scripted/harness run recorded in the WIP). (WP1 proved this via a harness replicating the plugin's `verify_signature`; WP2 keeps that guarantee, ideally as a reusable test where the seam allows.)

5. **A minimal temporary trigger affordance ships** — a small always-mounted "Check & update" control (replacing `UpdaterProbePanel`) that: on click runs `check()`; if an update exists, shows a **bare confirm dialog** (version + "Update & relaunch?"); on confirm runs download→install→self-clear→relaunch with inline progress/status text; on "no update" / error shows a status line. It is explicitly a **throwaway trigger** (WP4 replaces it with the polished non-modal notify / menu-item / skip-list UX) — clearly marked as such in code.

6. **The WP1 probe scaffolding is removed** — `src-tauri/src/updater_probe/` (module + `updater_probe_check`/`_run` commands + `mod updater_probe;` in `lib.rs` + the two `invoke_handler` entries) and `src/updaterProbe/` + its `App.tsx` import/mount are deleted, their durable pure-core migrated into the new `updater` module (AC-2). No dead `updater_probe` references remain (`cargo build` + `tsc` clean; a grep for `updater_probe`/`UpdaterProbePanel` returns nothing outside archived docs).

7. **All quality gates green:** `cargo build` + `cargo test` (incl. the migrated 6 self-clear tests) + `cargo clippy --all-targets -- -D warnings` (0 warnings), `pnpm tsc --noEmit`, `pnpm eslint`, `pnpm vite build`, and the frontend test suite all exit 0. No `unwrap()` outside tests; typed `thiserror` errors on the Rust boundary.

## Out of Scope (WP2)

- **The brew detect-and-defer gate (WP3).** WP2's flow always proceeds as a direct-download self-update. WP3 adds the `install_source()` gate that short-circuits brew installs. (WP2 should not hard-code any assumption that *prevents* WP3 slotting its gate in front of `check()`/notify — a clean seam, but no gate logic here.)
- **User-control UX + persistence (WP4).** No skip-this-version, no disable-notifications pref, no `config_store` update prefs, no non-modal notify, no app-menu "Check for updates…" item, no polished progress bar. WP2 ships only the minimal throwaway trigger.
- **The `/release` publishing pipeline (WP5).** WP2 does not change `/release`. It relies on the existing WP1-staged artifacts / a local manifest for verify; the real GH-published `latest.json` is a WP5 + WP6 concern.
- **The live installed-build Gatekeeper GO/FALLBACK verdict.** Deferred by the operator into WP6's real end-to-end verify (`SURFACE-2026-07-17-M10-WP1-LIVE-VERIFY-DEFERRED`). WP2 implements the **self-clear (GO-path) mechanism** at the proven seam; if WP6 later returns FALLBACK, WP4 adds the instruct-user dialog and WP2's flow is unchanged (the self-clear step becomes best-effort with the dialog as backstop). WP2 does NOT block on the live verdict.
- **Windows/Linux update paths.** macOS-only project; `darwin-aarch64` target only.

## Technical Constraints

- **3rd-party probe status: COMPLETE.** The 3rd-party surfaces (`tauri-plugin-updater`, GH-Releases endpoint, minisign) were de-risked by WP1 (the WBS §4 probe). Confirmed empirically: the artifact triad (`.app.tar.gz` + `.sig` + `.dmg`) emits under `createUpdaterArtifacts`; `latest.json` shape validates; minisign verify happens *inside* `download()`; a bad sig fails before install; macOS `install()` does NOT relaunch (clean self-clear seam); a user-owned `/Applications` bundle renames without an admin prompt. No known-unknown remains for WP2. (`arch.md` → "Milestone 10 architecture"; `research.md` M10; WP1 WIP findings P2.1–P2.4.)
- **The self-clear insertion point is FROZEN by WP1** (P2.1): after `install()` returns `Ok(())` on macOS, resolve the running app's own `.app` path, run `xattr -dr com.apple.quarantine <bundle>` (no sudo), THEN `relaunch()`. minisign verify is inside `download()` (also the cancel-safe boundary). Do not re-derive.
- **Durable pure-core migration, not rewrite.** `resolve_bundle_path` / `quarantine_clear_command` / `clear_own_quarantine` + their 6 tests are the shipping self-clear logic (WP1 verify-codify pinned them as durable). Move them into `src-tauri/src/updater/` intact; do not reimplement.
- **`tauri-command-removal-needs-invoke-sweep`** (memory): removing the `updater_probe_check`/`_run` commands requires a frontend `invoke()` caller sweep + a runtime smoke-launch — the FE/BE binding is stringly-typed and invisible to the unit gate. The new production commands need the same care.
- **Split `download()`/`install()`**, not `downloadAndInstall` — for the WP4 cancel boundary (research Trade-offs; arch "use the split download() then install()").
- **Rust relaunch = `app.restart()`** on the `AppHandle` (research confirmed); JS `relaunch()` from `@tauri-apps/plugin-process` is the alternative. WP1's probe used `app.restart()` — keep that.
- **Per bundle-identity** everything that touches per-install state respects `com.claudesk.app` vs `.dev` (established isolation) — though WP2 adds no persistence, its endpoint/config must not break dev/prod isolation.
- **WP5 signing gotchas already captured** (`SURFACE-2026-07-17-M10-WP5-SIGNING-KEY-ENV-VAR-IS-STRING-NOT-PATH`): env var is key *contents* not path; `latest.json` signature field is the `.sig` verbatim (already base64; don't re-encode). These bind WP5, noted here so WP2's local-manifest verify uses the correct shape.
- **`verify-self` posture (backend-lifecycle + destructive):** the production flow's terminal step (`install()`→replace→`relaunch()`) is destructive and Gatekeeper-dependent — inherently a WP6 installed-build observation. WP2 verify-self proves the agent-doable slice: compile/wire/type-check the flow, migrate + pass the self-clear tests, prove minisign REJECT, and confirm the trigger renders + the `check()` path is invocable (bridge or dev). The live install→relaunch is CARRIED to WP6, not driven in dev (driving `install()` in dev would replace the target/release binary — unsafe, zero signal).
