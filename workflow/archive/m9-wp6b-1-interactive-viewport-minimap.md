## Problem Statement

WP6a shipped the day-view timeline (`DayTimeline.tsx`) with a **fixed** viewport: the visible time window is computed once from the payload's `hour_range` (fallback `[6,23]`) and never changes ‚Äî no zoom, no pan, no way to inspect a dense burst of activity or see the whole day at once. The source `claude-time` dashboard had an **interactive zoom/pan viewport** (`ViewportContext` + `useTimelineGestures`) and a **Minimap** overview strip on top of it; WP6a deliberately dropped the entire subsystem to ship the MVP.

The operator has declared the **Minimap a MUST-HAVE, top priority above every other WP6b surface** (decision 2026-07-10). The Minimap is built entirely on the interactive viewport, so this WP has two coupled deliverables:

1. **Revive the interactive viewport** in `DayTimeline` ‚Äî replace the fixed `Viewport` prop with a shared, mutable viewport that responds to drag-pan, ctrl/cmd-wheel + trackpad-pinch zoom, and keyboard controls.
2. **Port the Minimap** ‚Äî a compressed all-projects density track with a draggable/resizable visible-window rectangle (drag body = pan, drag edges = zoom, click background = re-center) that reads and writes the *same* shared viewport.

This is a **capability** (interactive viewport), not a render port ‚Äî it reshapes `DayTimeline`'s fixed-viewport internals. Frontend-only over WP4's existing query layer + WP6a's shipped DayTimeline; **no new backend**.

**Problem statement unchanged (F12 back-loop re-check, 2026-07-13)** ‚Äî the root problem is still "revive interactive viewport + Minimap." Verify-human surfaced a sub-detail, not a shift: with `deriveDataWindow` returning the *same* range as the `hour_range` seed, the default-zoom viewport fills the whole data window, so `clampViewport` (correctly) has nowhere to pan ‚Üí drag felt dead. The capability is right; the data-window needs pan headroom. Fix is scoped, not a re-plan.

**Problem statement unchanged (2nd F12 back-loop re-check, 2026-07-13 ‚Äî chevron regression)** ‚Äî still "revive interactive viewport + Minimap." The Phase-3 verify-human regression (chevron no longer collapses) is another gesture-surface sub-detail: the pan handler's `onPointerDown` starts a drag + `setPointerCapture` on ANY non-seg target inside the timeline body, including the interactive chevron control ‚Äî capture then steals the chevron's click. The gesture design is right; its pan-exclusion guard was too narrow (only `[data-seg-id]`, not all interactive controls). One-line widen, not a re-plan.

## User Stories

- As the operator reviewing today's time analytics, I want to **drag-pan** the day timeline left/right so I can scan across the day without the fixed 6am‚Äì11pm window boxing me in.
- As the operator, I want to **ctrl/cmd-scroll or trackpad-pinch** to zoom into a dense burst of activity (e.g. a 20-minute window of rapid tool calls) so I can see individual segments that are 1px-wide at full-day zoom.
- As the operator, I want a **Minimap** overview strip showing the whole day's aggregate activity with a rectangle marking my current zoom window, so I never lose my place ‚Äî and I want to **drag the rectangle to pan**, **drag its edges to zoom**, and **click the background to jump** the viewport there.
- As the operator, I want the **hour ruler + grid to adapt their tick density** to the zoom level (1h ticks when zoomed out, 15m/5m/1m when zoomed in) so the timeline stays readable at any scale.
- As the operator, I want **clicking a segment to still select it** (the WP6a highlight ring), distinct from starting a pan ‚Äî clicking a segment must not drag the timeline.
- As the operator, I want the timeline, hour ruler, grid, and Minimap to **always agree** on the visible window ‚Äî one shared viewport, no drift.

## Acceptance Criteria

### AC-1 ‚Äî Shared viewport state (single source of truth)
- [ ] A **`ViewportContext`** (or equivalent single-owner state in `GlobalDashboard`/`DayTimeline`) holds one `{ visible_start_min, visible_end_min }` object in **minutes-from-local-midnight** (same unit as `SegPayload.start`/`.end` ‚Äî confirmed single-unit in source item 8). Values may be fractional during gestures.
- [ ] All viewport **reads** (`HourRuler`, `HourGridBackground`, `SegmentBar`, overlap layers, `Minimap`) resolve from this one source; all **writes** (gestures, keyboard, Minimap) go through one setter. No component holds a private copy of the viewport.
- [ ] The provider value is **memoized** so read-only consumers don't re-subscribe on unrelated re-renders (source `viz_render.py:854`).

### AC-2 ‚Äî WP6a fixed ‚Üí interactive migration (initial seed)
- [ ] The interactive viewport is **seeded** from the same source WP6a used: `viewportFromHourRange(data.hour_range)` (fallback `[6,23]` ‚Üí `{360, 1380}`). WP6a's `viewportFromHourRange` becomes the **initial state**, not a permanent fixed value.
- [ ] The `data-window` (pan/zoom bounds) is derived from the same `hour_range` via a `deriveDataWindow`-equivalent (single-day ‚Üí `[hour_range[0]*60, hour_range[1]*60]`; fallback `[0,1440]`). Single-day only ‚Äî the source's multi-day `day_count*1440` branch and all `dayOffset` math are **dropped** (source item 8: `dayOffset` is always 0 here).
- [ ] When the payload changes (new day fetch / tracking flip), the viewport **resets** to the fresh `hour_range` seed (no stale window carried across data changes).
- [ ] **All WP6a render behavior is preserved** under the interactive viewport: overlap markers/overlays (same-project-scoped), collapsed/expanded project rows, the selected-segment highlight ring, per-project totals, the 6-kind palette + `textOn` ink, and the `data-*` verify attributes (`data-seg-id`, `data-seg-kind`, `data-project-row`, `data-collapsed-seg`, `data-overlap-*`). Nothing WP6a shipped regresses.

### AC-3 ‚Äî Clamping (a REAL clamp ‚Äî the source has none)
- [ ] A pure **`clampViewport(vp, dataWindow, minSpanMin)`** exists and **every** viewport write routes through it ‚Äî drag-pan, wheel-zoom, keyboard, AND Minimap pan/edge/recenter. (Source finding item 3: the reference leaves pan *unbounded* and has no clamp function ‚Äî this is a deliberate improvement, not a port gap.)
- [ ] **Max zoom-out:** visible span cannot exceed the data-window width (`dataWindow[1]-dataWindow[0]`).
- [ ] **Max zoom-in:** visible span floors at a minimum (`minSpanMin`, lean **1 minute** matching source wheel/edge floors) ‚Äî never inverts (`visible_start_min < visible_end_min` always).
- [ ] **Pan containment:** the viewport cannot pan so its window leaves the data-window; clamping a pan **preserves width** (shift both endpoints, then translate back into bounds) rather than shrinking at the edge. (Explicitly better than the source, which lets pan run off the data.)
- [ ] **Degenerate guard:** when `dataWindow` width `<= 0` (empty/one-instant day), the timeline + Minimap render without a divide-by-zero / 0-width crash (source item 6 has NO such guard ‚Äî this is a required fix). Extreme zoom (1-minute span across a full render width) does not crash or produce `NaN`/`Infinity` geometry.

### AC-4 ‚Äî Timeline gestures (`useTimelineGestures` port)
- [ ] **Drag-pan:** pressing on empty timeline background and dragging pans the viewport. `pointerdown`‚Üícapture‚Üí`pointermove` deltas convert px‚Üíminutes against the *frozen* start-viewport (so RAF-coalesced frames are idempotent, source item 4). Drag right ‚Üí content moves right (pan toward earlier). Cursor shows `grab`/`grabbing`.
- [ ] **Wheel/pinch zoom:** `ctrl` OR `cmd`(meta) + wheel zooms; **plain wheel does nothing to the viewport** (page-scroll behavior preserved). Trackpad pinch is covered by the same `ctrlKey||metaKey` gate ‚Äî **no separate gesture handler** (browsers synthesize `ctrlKey` on pinch, source item 4). Zoom is **cursor-anchored**: the data-minute under the cursor stays under the cursor across the zoom. Zoom factor ~1.1/tick.
- [ ] **Click-vs-pan disambiguation:** replicate the source's DOM-target mechanism (item 4) ‚Äî the pan `pointerdown` **early-returns when `e.target.closest('[data-seg-id]')` is truthy**, so a mousedown on a `SegmentBar` never starts a pan and its `onClick` select fires normally. **No pixel/distance threshold.** A click on background is a zero-delta pan (harmless).
- [ ] **RAF throttling:** at most one `setViewport` per animation frame (cancel-and-reschedule; latest wins). **The pending RAF is cancelled on unmount** (source LEAKS this ‚Äî required fix).
- [ ] Listeners are React synthetic handlers on the timeline wrapper element (auto-cleanup); only a `window` keyboard listener (if AC-6 adopted) needs explicit teardown.

### AC-5 ‚Äî Adaptive tick density (`pickTickInterval`/`ticksInViewport` port)
- [ ] `HourRuler` and `HourGridBackground` recompute tick interval from the **live viewport width** each render via a `pickTickInterval` port: pick the densest interval from `[1440,360,60,30,15,10,5,1]` minutes whose tick count lands in the **8‚Äì30** band (source item 5). Extreme zoom-in ‚Üí 1m; extreme zoom-out ‚Üí 1440.
- [ ] `ticksInViewport` generates minute-aligned ticks from the first aligned position ‚â• `visible_start_min` up to (exclusive) `visible_end_min`; single-day labels are `HH:00` (60m interval) or `HH:MM` (finer). The multi-day day-label / midnight-crossing branches are dropped (single-day, `windowStartIso=null`).
- [ ] WP6a's fixed `hoursInViewport`/hardcoded `HH:00` ruler is replaced by the adaptive version. The ruler stays readable (no label pile-up) at full-day-out and 1-minute-in extremes.

### AC-6 ‚Äî Keyboard controls (SCOPE DECISION ‚Äî see Open Questions Q4)
- [ ] *(If in scope)* Arrow-pan (¬±10% range), `+`/`-` zoom, `0` reset-to-seed, Home/End jump-to-edge (preserving range) via a `window` keydown listener that ignores INPUT/TEXTAREA/contentEditable focus and is torn down on unmount. All writes go through `clampViewport`. *(Resolve Q4 before build; if deferred, drop this AC and log a SURFACE.)*

### AC-7 ‚Äî Minimap (`Minimap` port ‚Äî the priority deliverable)
- [ ] Renders an **all-projects aggregate density track** (every segment from every project, flattened) as absolutely-positioned colored bars over the full `dataWindow` extent, `pointerEvents:'none'` so bars don't intercept drags, ~0.6 opacity, `Math.max(0.1%, widthPct)` min-width so tiny segs stay visible (source item 6). Colored by the 6-kind palette (`segStyle`/`colorForKind`).
- [ ] A **visible-window rectangle** whose left%/width% = `(viewport - dataWindow[0]) / dwWidth` ‚Äî reads the shared viewport, so it tracks every timeline gesture live.
- [ ] **Drag rectangle body = pan** (shift both endpoints by px‚Üíminutes across the full window; width preserved).
- [ ] **Drag rectangle edges = zoom:** two **6px** grab strips straddling each edge (`ew-resize` cursor); edge-left moves only `visible_start_min`, edge-right moves only `visible_end_min`, each with the 1-minute min-span floor. Hit-tested by `data-minimap-mode` on the target (source item 6), not geometric distance.
- [ ] **Click background = re-center:** centers the viewport on the clicked data-minute, **preserving current range**; `clientX‚ÜídataMin` clamps `frac` to `[0,1]`. Fires on mousedown; `stopPropagation` so the timeline's gesture handler doesn't double-react.
- [ ] Minimap writes go through the **same** `setViewport` + `clampViewport` as the timeline (shared viewport, AC-1/AC-3). Its own RAF is **cancelled on unmount**. It guards `dwWidth <= 0` (no divide-by-zero ‚Äî source lacks this).

### AC-8 ‚Äî Shared sync (the whole point)
- [ ] Any gesture on the timeline (drag/wheel/pinch/keyboard) moves the Minimap rectangle in the same frame, and any Minimap gesture (pan/edge/recenter) moves the timeline segments + ruler in the same frame. No drift, no lag beyond one RAF. Verified live (AC below / verify-self).

### AC-9 ‚Äî Verification (live, MCP bridge)
- [ ] Live-verified via the `tauri` MCP bridge against a **scratch workspace** (`tmp/scratch/scratch-{a,b,c}`) with tracking ON and a driven CC turn generating real data, dashboard opened via `‚åò‚áßA`:

  - ctrl/cmd-wheel zoom (and, where drivable, pinch) zooms cursor-anchored;
  - Minimap rectangle body-drag pans, edge-drag zooms, background-click recenters;
  - clicking a segment still fires the highlight ring (select), does NOT start a pan;
  - tick density visibly adapts between zoomed-out and zoomed-in;
  - **no 0-width / reflow / NaN crash** at extreme zoom or on an empty/degenerate day.
  - *(Fidelity caveat per CLAUDE.md: raw xterm typing is low-fidelity via the bridge, but pointer/wheel/DOM/screenshot are high-fidelity ‚Äî the gesture drives here are pointer/wheel, which the bridge handles. ## Retrospect
- **What changed in our understanding:** The source `claude-time` viewport subsystem was less complete than the WBS line-number references implied ‚Äî the *real* interactive state provider lived in `viz_render.py`, not `dashboard.jsx` (which was a design-canvas prototype with context-defaults only), and there was **no `clampViewport` at all** (pan was left unbounded). - **Assumptions that held:** The pure-math-core + thin-DOM-glue + live-bridge-verify architecture (project convention) held perfectly ‚Äî 56 pure vitest pins caught every math edge, and the bridge drove every gesture except raw pointer-drag *feel*.
