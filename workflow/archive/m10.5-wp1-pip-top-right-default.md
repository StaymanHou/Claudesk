## Problem Statement

On first summon, the PiP NSPanel opens at NSPanel's default origin (~screen-center). The operator has to drag it to the top-right corner every session — a daily-driver papercut (`SURFACE-2026-07-16-M10.5-QOL-BUCKET` item 1). Fix: on first show, anchor the panel near the **active screen's top-right corner** (computed from the screen's `visibleFrame`), reusing the proven main-thread `setFrameOrigin:` AppKit path already in `pip_move`. A later operator drag must be **preserved** — the top-right anchor applies **only when the panel has no prior position this session**, never re-anchoring on every summon (that would fight the drag).

[Updated 2026-07-18: operator requested a **150px inset from the top + right edges** (confirmed "Top + right" via AskUserQuestion at verify-self), NOT flush against the corner. So: `right edge = visibleFrame.maxX − 150 − panelWidth`, `top edge = visibleFrame.maxY − 150 − panelHeight` (bottom-left/y-up). The flush-anchor build's mechanics — anchor-in-`pip_resize`, `positioned`-flag drag-preservation — PASSED live verify-self; only the margin changes.]

## Key code-seam facts (verified against current source, 2026-07-18)

- **`pip_set_visible` builds the panel at a placeholder size** (220×130, `commands.rs` ~L309) — the *real* content-driven size arrives **asynchronously**: the PiP webview (`Pip.tsx` ~L180) computes size from layout × workspace-count × screen and applies it via the backend `pip_resize` command (`set_content_size`) within the first frame or two. **Consequence:** anchoring at build-time would anchor against the 220×130 placeholder, landing the panel in the wrong spot after the resize. The anchor MUST run against the **current (post-resize) frame size**, so the natural seam is **inside `pip_resize`** (after `set_content_size`), not in the build branch of `pip_set_visible`.
- **`pip_move` (`commands.rs` ~L179) is the ONLY path that moves this swizzled panel** — a raw `msg_send![ns_panel, setFrameOrigin:]` on the main thread. NSWindow frame origin is **bottom-left, y-up**. Top-right = `origin.x = visible.maxX − panelWidth`, `origin.y = visible.maxY − panelHeight`. WP1 reuses this exact AppKit call.
- **Drag position is NOT persisted** — `pip_move` mutates the live panel in-session only; there is no stored origin, and on relaunch the panel rebuilds fresh. So "preserve a later drag" is an **in-session flag**, NOT persisted settings. Re-anchoring top-right on each fresh launch is correct (matches the operator's stated preference; no need to persist a per-launch drag).
- **Shared mutable flag home:** `PipAutoStateLock(pub Mutex<PipAutoState>)` in `pip/mod.rs` is already Tauri-managed. Add a `positioned: bool` field to `PipAutoState` — the anchor reads/sets it under the existing mutex. No new managed type. It resets to `false` on each app launch (default) → fresh top-right anchor per launch, as intended.
- **Command bodies run on the main thread** (`pip_resize`, `pip_move` are `#[tauri::command]`), so the `setFrameOrigin:` call inside `pip_resize` is main-thread-safe with no marshaling — same as the existing `pip_move` and `set_content_size` calls. (The CLAUDE.md main-thread rule only bites background-thread callers; `pip_resize` is not one.)

## Design note

No design prior fires on the *anchor choice* — top-right is the operator's stated bare preference, not a product-design lean. The one real design decision is the **"anchor only when unpositioned, preserve drag"** rule: it keeps the panel predictable without introducing a mode or a persisted position. `pip_move` sets `positioned = true`; the anchor is a no-op once positioned. `pip_set_mode`/toggle re-show does NOT reset `positioned` (re-summoning a panel you already dragged keeps it where you left it, this session).

## Retrospect
- **What changed in our understanding:** Two things the plan didn't anticipate. (1) **The anchor seam is `pip_resize`, not the build branch.** The plan assumed anchoring "at the first-show seam in `pip_set_visible`", but reading the code showed `pip_set_visible` builds at a 220×130 *placeholder* and the real content-driven size lands asynchronously via the webview's mount-time `pip_resize` — so anchoring at build time would have used the wrong size. The correct seam is inside `pip_resize` (after `set_content_size`), which also gives the layout-recompute behavior for free. (2) **Drag position is not persisted** — `pip_move` mutates the live panel only, no stored origin — so "preserve a later drag" is an in-session `positioned` flag, not persisted state, and a fresh launch re-anchors (which matches the operator's preference).
- **Assumptions that held:** The `setFrameOrigin:` AppKit path from `pip_move` was reusable verbatim; both mutation sites are `#[tauri::command]` bodies (main thread) so no `run_on_main_thread` marshaling was needed; the pure-math extraction (`top_right_origin`) was cleanly unit-testable; the MCP bridge (`manage_window info {windowId:'pip'}`) reads native NSPanel geometry directly, making agent-driven verify-self of a native-window feature possible.
- **Assumptions that were wrong:** The plan (and the roadmap/WBS) framed the anchor as **flush** to the top-right corner. At verify-self the operator asked for a **150px inset** from the top+right edges instead — one F9b back-loop added a `margin` param + `PIP_ANCHOR_MARGIN=150` const. Cheap change (the math was already parameterizable), but a reminder that "top-right" had an unstated gap preference.
- **Approach delta:** Implementation matched the plan's *structure* (pure helper + gated anchor + flag) but relocated the anchor from `pip_set_visible` → `pip_resize` (delta 2 above) and changed the anchor from flush → 150px inset (operator F9b). Both deltas are recorded as AS-BUILT notes in `docs/product/wbs.md` WP1. One extra verify cycle (F9b) vs. the single-pass plan; otherwise clean.
