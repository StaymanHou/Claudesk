## Problem Statement

WP2 lifted the N=1 clamp: N projects now coexist as mounted workspaces, the center stage switches to whichever is focused, and the (M1-empty) filmstrip slot carries only a "+" control. But **the only way to switch back to a backgrounded workspace is to re-open it via the "+" picker** — there is no roster of open workspaces and no in-place switch affordance. The user also can't see at a glance which workspaces are open, what each is running, or which one holds the center stage.

WP3 populates the filmstrip slot with the actual multi-workspace surface: **one tile per open workspace** (including a static, active-marked tile for the center-staged one), each carrying a project-name label + an **M3-hook-driven status dot**; **background tiles render a live ~1 fps `serializeAsHTML()` terminal mirror** (the WP4-probe-validated path); and a workspace is promoted to center stage by **clicking its tile OR pressing `⌘⇧+digit`**. Tiles are **drag-reorderable**, and that persisted order is the single source of truth for both render order and the `⌘⇧+digit` index.

This is the surface that makes "find the project waiting on input" a glance + one click — the second-order tax the vision calls out — and it closes `SURFACE-2026-06-23-WP3-BACKGROUND-WORKSPACE-SWITCH-AFFORDANCE` (the tile-click / `⌘⇧+digit` switch is WP3's **primary** path, verified explicitly at verify-human).

## User Stories

- As the operator, I want **one tile per open workspace** in the filmstrip — including the focused one — so the row is a complete, stable roster of what's open.
- As the operator, I want each tile to show a **status dot** (idle / running / awaiting-input) driven purely by the M3 hook channel, so I can spot the workspace waiting on me without clicking through.
- As the operator, I want **background tiles to show a live ~1 fps mirror** of their terminal, so I can see *what* a backgrounded session is doing, not just its status.
- As the operator, I want to **click a tile to promote that workspace to center stage** (demoting the current one), without any unmount/remount — PTY + panel state survive the switch.
- As the operator, I want **`⌘⇧1`…`⌘⇧9`** to jump to the Nth tile from anywhere (inside the CC terminal, inside CodeMirror, anywhere) — the keyboard equivalent of clicking it.
- As the operator, I want to **drag tiles to reorder** the filmstrip, with the order persisting across restarts and being what `⌘⇧+digit` indexes into, so my muscle memory for "⌘⇧2 = project X" stays stable.

## Acceptance Criteria

The feature is done when:

1. **Roster completeness + order.** The filmstrip renders exactly one tile per open workspace, **including the center-staged one** (a static active-marked tile, no mirror), in the user-arranged persisted order. The "+" add-workspace control (WP2) remains. Tile count tracks the WorkspaceList: opening appends a tile, the roster never silently drops or duplicates a workspace.
2. **Status dots agree with the header.** Each tile's status dot subscribes to the same M3 `workspace-status` event (keyed by `workspace_id`) via the existing `useWorkspaceStatus` + `WorkspaceStatusIndicator` palette (Idle / Running `#d97757` / AwaitingInput `#539bf5` / Unknown). The filmstrip dot and the center-stage header dot for the same workspace are always identical (single broadcaster fans out to both).
3. **Live background mirror.** Each **background** tile shows a ~1 fps mirror built from `@xterm/addon-serialize` `serializeAsHTML()` read off that workspace's still-mounted-but-off-viewport xterm buffer, rendered into a `scale(...)` tile. The center-stage tile is a **static placeholder** (name + dot + active marker, NO mirror — it's already full-size on stage).
4. **Click-to-promote.** Clicking a background tile makes that workspace center stage (`display:block`) and demotes the previous (`display:none`) — **no unmount/remount**; PTY + panel state persist (M1 rule). The newly-promoted tile flips to the static placeholder; the demoted tile starts its mirror loop. Clicking the already-active tile is a no-op.
5. **`⌘⇧+digit` promote.** `⌘⇧1`…`⌘⇧9` promote the Nth tile (in the persisted order, including the center-stage tile so indices are stable) to center stage. Fires regardless of focus (capture-phase document listener, `preventDefault`). N past the tile count → no-op (or clamp to last — decided at plan, mirroring WP12's past-end behavior). Disjoint from bare `⌘+digit` (the M2 editor tab-switch).
6. **Drag-to-reorder, persisted.** Tiles are drag-reorderable; the new order persists (localStorage, consistent with M2 panel-width/collapse patterns) and is the single source of truth for both render order and the `⌘⇧+digit` index. The center-stage tile participates in the order (stable slot). Reorder survives an app restart.
7. **Serialize-loop lifecycle.** The ~1 fps loop runs per background tile, stops when a tile becomes center stage (mirror → placeholder), and is structured so WP4's collapse can stop it wholesale. No loop runs for the center-stage tile. (Honor any WP1 lazy-mount consequence — see Technical Constraints; WP1's verdict was GO for eager-mount, so all workspaces are mounted and serializable.)
8. **Tests (vitest).** Pure tile-list derivation (WorkspaceList + persisted order → ordered tiles, center-stage marked-not-excluded); status-dot mapping reuse; `workspaceSwitchIndex` chord-index mapping; reorder-persistence reducer; serialize-throttle timer logic if extractable as a pure seam. Existing suites stay green (vitest 361 baseline, cargo 186, clippy -D warnings clean).
9. **verify-human (live, native app).** With ≥2 real CC sessions open: tiles show live mirrors for backgrounds + static active tile; status dots flip on real idle→running→awaiting-input; click-a-background-tile AND `⌘⇧+digit` both promote correctly (no remount, PTY survives); drag-reorder persists across a restart; `⌘⇧+digit` still hits the right tile after a reorder. **Explicitly verify the background-workspace-switch affordance** (`SURFACE-2026-06-23-WP3-BACKGROUND-WORKSPACE-SWITCH-AFFORDANCE`).

## Out of Scope

- **Collapse toggle** (expanded thumbnails ↔ collapsed status-pill row) — that's **WP4**; WP3 builds the expanded surface only. WP3 must structure the serialize loop so WP4 can stop it wholesale, but ships no collapse UI.
- **Left/right intra-workspace focus indicator** — **WP4b** (parallel track).
- **Menu-bar / PiP status surfaces** — M5/M6; they subscribe to the same M3 broadcaster but are not built here.
- **PiP-style click-to-focus from a non-Claudesk surface** — N/A here.
- **Pre-optimizing the active-CPU p95** — the WP4 probe flagged ~30% p95 on output bursts (median ~13%). Documented mitigations (sub-1fps background rate, coalesced serialize, mirror-only-visible) are available *if dogfooding shows it bites* — note, don't pre-optimize (WBS task).
- **Building on the WP1 throwaway probe code** (`src/probe/nworkspaces/`, the `?nwsprobe` branch in `src/main.tsx`) — DEV-only, deleted at M4 `/product-finalize`. WP3 must NOT depend on it.

## Technical Constraints

**Validated seams to reuse (do not reinvent):**
- **Status:** `src/state/useWorkspaceStatus.ts` (`stateFor(workspaceId)`) + `src/components/workspace/WorkspaceStatusIndicator.tsx` + the `workspaceStatus.ts` presentation palette. `App.tsx` already threads `statusFor={stateFor}` into CenterStage; pass the same down to the Filmstrip so both surfaces agree.
- **Workspace list:** `src/state/workspace.ts` reducer + `useWorkspaceList.ts`. WP2 already provides `workspaces: Workspace[]`, `focusedId`, `focusWorkspace(id)`. Promotion = call `focusWorkspace(id)` (it sets `focusedId`; CenterStage's `display:none`/`block` toggle keys off `focusedId === w.id`). **`focusWorkspace` is the promote primitive — both click and `⌘⇧+digit` route through it.** (Note: `useWorkspaceList` does not currently expose `focusWorkspace` in its returned API — it's defined but the return omits it; App.tsx destructures only `workspaces, focusedId, view, openWorkspace, setSessionId`. WP3 must surface `focusWorkspace` through the hook API.)
- **Mount model:** all workspaces stay mounted; CenterStage toggles `display`. WP1 verdict = **GO for eager-mount** (`docs/product/wp1-n-workspace-cost-probe-outcome.md`), so every workspace's xterm is mounted and its buffer is serializable. Background workspaces must be **off-viewport (`left:-99999px`), not `display:none`** — the WP4 probe found `display:none` breaks FitAddon AND it's what lets xterm's `IntersectionObserver` pause the off-screen renderer for free while `write()` still updates the buffer. **Confirm/adjust how CenterStage currently hides background workspaces** (`display:none` vs off-viewport) — this is a likely WP3 touch-point, since live serialize needs the buffer fed but the renderer paused.
- **Mirror mechanism:** `@xterm/addon-serialize` `serializeAsHTML()` from the buffer, throttled to ~1 fps, into a `scale()` tile. NOT `cloneNode`, NOT off-screen-DOM-mirror. Pauses on `document.hidden`. (WP4 probe, arm B — the recommended arm.) **Check whether `@xterm/addon-serialize` is already a dependency** (the probe harness used it; confirm it's in `package.json` proper, not just probe-scoped).
- **`⌘⇧+digit` chord:** RESERVED for exactly this in `src/components/workspace/editor/paletteCommands.ts` (chord-ownership map, lines ~48–49) — memory `cmd-shift-digit-reserved-for-filmstrip`. Build a pure predicate **`workspaceSwitchIndex(e)`** mirroring `editor/tabSwitchChord.ts`'s `tabSwitchIndex` shape (returns 1-based digit or null; requires `metaKey && shiftKey`, key 1–9). Register on the **capture-phase document listener** (the WP1-proven app-level pattern — fires regardless of focus, `preventDefault` so CM6/terminal don't swallow it). Bare `⌘+digit` is the M2 editor tab-switch — disjoint (Shift present vs absent).
- **Persistence pattern:** localStorage, consistent with M2 panel-width/collapse persistence (find the existing M2 localStorage helper/key convention and follow it; e.g. filetree `railWidth.ts`, editor `fontZoom.ts`). Order is an array of workspace ids (or a stable per-project key — decide at plan; ids are minted per-session `ws-N`, so **persisting by `project_path` is likely more durable across restarts than by ephemeral `ws-N` id** — flag for plan).

## Open Questions — RESOLVED at plan (2026-06-23)

- [x] **Persist order by `project_path`, NOT by `ws-N` id.** `ws-N` ids are minted per session (monotonic counter, reset on restart), so an id-keyed order won't survive a restart. The persisted order is an **array of `project_path` strings**; on load it's mapped back to the live workspaces (open paths render in stored order; any open path not in the stored list appends at the end in WorkspaceList order; stored paths no longer open are ignored). This mirrors the M2 `railWidth.ts` / `fontZoom.ts` localStorage posture (pure load/save helpers, app-global UI chrome, swallow storage errors).
- [x] **`⌘⇧+digit` past-end = NO-OP** (not clamp). The roster is complete — every open workspace has a tile — so `⌘⇧N` where N > tile-count means "no such tile" and is a clean no-op. (WP12's editor tab-switch *clamps* because its strip length is incidental; here N maps 1:1 to a known roster, so out-of-range is genuinely nothing to switch to.) `workspaceSwitchIndex` returns the 1-based digit; the promote handler returns early when `index > tiles.length`.
- [x] ~~Drag-reorder = native HTML5 drag-and-drop~~ **REVISED 2026-06-23 (P4 verify-human, F23):** native HTML5 DnD was wrong on TWO counts — (1) `onDrop` did not reliably fire in the WKWebView (`<button draggable>` + nested mirror content + dataTransfer quirks → drag worked, drop no-op'd), and (2) HTML5 DnD **cannot do WYSIWYG live-reorder** (it only exposes dragover/drop, not on-the-fly repositioning) which the operator now requires ("things reorder as I drag, before I drop"). **New approach: POINTER-based live reorder** (`pointerdown`/`pointermove`/`pointerup` + `setPointerCapture`): on each move, hit-test the pointer's x against tile midpoints and reorder LIVE (the order state updates every move, so tiles shuffle in real time); commit + persist on pointerup. The pure `reorder(items, from, to)` helper is reused unchanged; only the event glue changes. No new dependency (still vanilla).
- [x] **Serialize loop = ONE shared ~1 fps ticker** iterating all *background* tiles, not per-tile timers. Cheaper, single `document.hidden` gate, and WP4's collapse can stop it with one teardown. The center-stage tile is skipped each tick (it's a static placeholder).

## Retrospect
- **What changed in our understanding:** The hard problems in WP3 were NOT the planned ones (tile derivation, status dots, chord, persistence — all landed clean first try). They were the *live-rendering* details the plan under-weighted: (1) `serializeAsHTML()` is clipboard-oriented — written into a narrow node its `<pre>` reflows into a vertical white bar (needs a fixed terminal-natural width + scale); (2) a backgrounded terminal's serialized **viewport is parked** — `scrollback:0` froze the mirror once output passed one screen, fixed by a bottom-anchored `scrollback:40` tail; (3) **pointer capture on a per-tile button dies the instant a live reorder re-renders it** — the WYSIWYG drag's root cause, only found by building a throwaway hello-world + driving it with a real Playwright mouse.
- **Assumptions that held:** the pure-core/impure-shell decomposition; reusing `useWorkspaceStatus`/`WorkspaceStatusIndicator` for the dots (filmstrip + header agree for free); the `terminalMirror` registry seam (no prop-drill); `focusWorkspace` as the single promote primitive for both click and chord; persist-by-project_path surviving the ws-N id reset; ⌘⇧+digit reserved-binding being collision-free.
- **Assumptions that were wrong:** (a) native HTML5 DnD as the drag mechanism (Q3) — `onDrop` doesn't fire in the WKWebView AND it can't do WYSIWYG live-reorder → F23-revised to pointer events; (b) the off-viewport mount switch was treated as a clean swap, but it leaks background workspaces into the tab order / a11y tree (the `inert` MINOR finding); (c) "verify it in the browser is enough" for the mirror — the headless browser has no PTY, so the live-tail + dark-theme + overlay all needed native-app verify-human rounds.
- **Approach delta:** Phases 1, 2, 4-core matched the plan. Phase 3 (mirror) took 3 in-place operator-driven fixes (white-bar / tail / dark+overlay). Phase 4 (drag) took a full mechanism rebuild (HTML5→pointer) + 2 more fixes (left/right symmetry, capture-on-stable-strip) — the latter resolved only after the operator directed building a standalone test harness first.
