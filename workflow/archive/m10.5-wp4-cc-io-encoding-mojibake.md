## Problem Statement
Garbled characters (`�`, e.g. the operator-observed `-�.`) appear in CC input and/or
output inside a Claudesk workspace terminal. Expected: multi-byte glyphs (accented
chars, emoji, box-drawing, arrows) round-trip cleanly both when typed/pasted INTO the
CC prompt and when rendered FROM CC's output.

⚠️ **The SURFACE's original prime suspect is already mitigated.** The FE output path
already writes **bytes** (`decodeBase64` → `Uint8Array` → `term.write(bytes)` in
`XtermPane.tsx`), so the "split multi-byte char decoded to a string then written"
hypothesis is NOT the bug. Reproduce-first was doubly load-bearing here — a blind fix
would have targeted the already-correct layer.

## Reproduction Attempt

**Surface chosen:** failing test (pure-function, deterministic) + empirical env probe
**Outcome:** reproduced — **TWO distinct root causes**, both confirmed
**Determinism:** every-run (both)

### Root cause A — INPUT side truncation (the failing test anchors this)
`encodeBase64` in `src/cc/bridge.ts` does `String.fromCharCode(data.charCodeAt(i) & 0xff)`
per code unit. Any code unit > 0xFF (accented chars, arrows, €) or a surrogate pair
(emoji) is **truncated to one byte**, so CC never receives the glyph's real UTF-8 bytes.

Empirically reproduced (node, same JS engine as vitest):
```
é      correct-utf8=[195,169]      bridge-sends=[233]      CC-sees="�"   *** CORRUPT ***
€      correct-utf8=[226,130,172]  bridge-sends=[172]      CC-sees="�"   *** CORRUPT ***
café   correct-utf8=[99,97,102,195,169]  bridge-sends=[99,97,102,233]  CC-sees="caf�"
🎉     correct-utf8=[240,159,142,137]  bridge-sends=[60,137]  CC-sees="<�"  *** CORRUPT ***
→      correct-utf8=[226,134,146]  bridge-sends=[146]      CC-sees="�"   *** CORRUPT ***
```

**Artifact:** `src/cc/__tests__/bridge.test.ts` →
`"encodeBase64 sends a glyph's real UTF-8 bytes (multi-byte input round-trips)"` — RED.
Fails with `expected [233] to deeply equal [195, 169]`. The other 9 tests pass.

- **Fix direction:** encode the string's real UTF-8 bytes (`new TextEncoder().encode(data)`
  → base64) so the input round-trips. `decodeBase64` is already correct (returns bytes).
- **Plan note — a pre-existing test will need updating.** `"encodeBase64 masks to a single
  byte per char"` (bridge.test.ts ~L29) asserts the *buggy* `& 0xff` contract; the fix
  changes the contract from "mask" to "UTF-8 bytes", so that test's expectation changes.
  Not a reproduce concern — flagged for the plan.

### Root cause C — OUTPUT side locale unset in the installed `.app`
The CC/shell spawn env sets only `TERM=xterm-256color` + `COLORTERM=truecolor`
(`cc_session/mod.rs` `color_tty_env()`), **no `LANG`/`LC_*`**. `CommandBuilder` inherits
the parent env. A Finder/Dock-launched `.app` runs under the **minimal launchd env**,
where `LANG` is **unset** → the spawned `claude`/shell/child TUI defaults to
`LC_CTYPE=C` (ASCII/POSIX), which mangles UTF-8 **output** as `�`.

Empirically confirmed:
```
launchctl getenv LANG                         → (empty / unset)  ← the installed .app's env
env -i PATH=... /usr/bin/locale → LC_CTYPE="C" LANG=""            ← bare-launchd result
/bin/zsh -l -i -c 'echo $LANG'  → LANG=C.UTF-8                    ← what tauri:dev inherits
```

**Why it hid in dev:** `pnpm tauri:dev` inherits the launching terminal's login-shell env
(`LANG=C.UTF-8` — UTF-8-capable), so output-mojibake never reproduces there. The installed
build gets no `LANG` at all → `C` locale. This is the exact "passes in tauri:dev, breaks
in the installed `.app`" class the CLAUDE.md installed-build-smoke-test convention warns
about. (Note: even dev's `C.UTF-8` is not a region locale — the fix should set a concrete
UTF-8 locale like `en_US.UTF-8` in the spawn env so dev and install agree.)

- **Fix direction:** set an explicit UTF-8 locale in the spawn env
  (`LANG=en_US.UTF-8`, and likely `LC_ALL`/`LC_CTYPE`) in `color_tty_env()` or the
  spawn-env assembly, so both spawn paths (CC + WP9 shell) get a UTF-8 `LC_CTYPE`
  regardless of how Claudesk was launched.
- **Verify at:** the installed `.app` (per convention — this bug ONLY manifests there).

### Suspect B — output chunk-boundary — INVESTIGATED, NOT the cause
`spawn_reader_thread` base64-encodes each 4096-byte `READ_CHUNK` independently
(`mod.rs:556`), so a UTF-8 char CAN split across two `cc-output` events. But the FE
writes **bytes** to xterm (`term.write(Uint8Array)`), and xterm's decoder carries its
partial-UTF-8 state across separate `write()` calls — so a boundary split reassembles
correctly on render. Not the observed bug; left as-is. (If a future case shows xterm
NOT coalescing, a Rust-side partial-UTF-8-tail buffer across reads is the fix — deferred.)

## Fix sizing (for the plan)
Two independent, small, mechanical fixes over proven seams:
- (A) `encodeBase64` → `TextEncoder`-based UTF-8 encoding + update the one stale
  assertion. Pure FE, unit-testable. ~S.
- (C) add UTF-8 locale to the spawn env. Pure BE, one env addition shared by both
  spawn paths. ~S. Verify on the installed `.app`.

Neither introduces a new data model, endpoint, arch decision, webview, or dependency.
Combined they fit the small/simple criteria → **F33 → `/feature-plan`**.
