## Problem Statement

The M9 time-analytics dashboard ships the per-session time-shaped surfaces (Day/Week/Month timelines + SidePanel) and — as of WP6c-1 — the window-level **aggregate metrics** (Metrics tab: HeadlineCard + MetricsPanel over a single window). What's still missing is the **A/B comparison** that makes those aggregate numbers *legible over time*: "this week vs. last week, this month vs. last, today vs. a trailing baseline" — the trend view that tells the operator whether AI-leverage / parallelism / blocking-split are moving up or down. claude-time computed this via `build_comparison_data` + three preset helpers; Claudesk's WP4 query layer explicitly deferred it (`query.rs:35-36`). WP6c-2 builds the Rust producer (reusing `build_metrics` from WP6c-1 per A/B side, **skipping the dead `deltas` map** — CompareView recomputes deltas FE-side) and the dark-TSX Compare tab (CompareView + 8 EffectivenessRows + PresetSelector), enabling the currently-disabled Compare tab. It also closes the **last** piece of `SURFACE-2026-07-06-M9-COLOR-FAMILIES-AI-VS-HUMAN` (PARTIALLY-RESOLVED) — the 2 EffectivenessRow color-family bar blocks (blocking-split + concurrency-mix) routed through the existing `familyOf`/`colorForKind` seam rather than the source's hardcoded `CT_TOKENS.active`/`textMuted`.

## Resolved decisions (from the archived WP6c spec + this plan's context read)

- **D1 — `time_analytics_query` extension shape (deferred-to-plan #1, RESOLVED).** Follow the WP6c-1 `Metrics` precedent exactly. Add a `QueryWindow::Compare { spec }` variant (`#[serde(tag="kind", rename_all="snake_case")]` → `{"kind":"compare", "spec": …}`). `spec` is a nested tagged enum `CompareSpec`: `{"preset":"wow"|"mom"|"today_vs_trailing"}` OR `{"custom":{"a":{"start_ms","end_ms"},"b":{"start_ms","end_ms"}}}`. The command resolves each side's inclusive local day-bounds Rust-side (preset math mirrors the Python helpers), calls `build_metrics` per side, returns `ComparisonPayload`. **No new command** — extends `time_analytics_query`; result adds `TimeAnalyticsResult::Compare(ComparisonPayload)`.
- **D2 — filter-chip projection (deferred-to-plan #2, RESOLVED → DEFER).** The source CompareView projects each side through `_computeMetricsView(…, filterKinds)` from `useFilter()`. Claudesk's Metrics tab (WP6c-1) built **no** kind-filter seam, and CompareView is the only comparison consumer — so feed `comparison.a.metrics`/`comparison.b.metrics` straight through. Drop `_computeMetricsView` entirely. (If a kind-filter surface is ever added to Metrics/Compare, this is a clean follow-on; not built now — no unconsumed contract.)
- **D3 — `ComparisonPayload` shape (skip `deltas`).** `{ a: CompareSide, b: CompareSide, meta: ComparisonMeta }` where `CompareSide = { metrics: MetricsPayload, range: { start, end, day_count } }` and `ComparisonMeta = { a_start, a_end, b_start, b_end, a_day_count, b_day_count }`. CompareView reads `a.metrics`/`b.metrics` + `meta` only (confirmed by reading `dashboard.jsx:1271-1410`). **No `deltas` map** — every delta is recomputed FE-side from the two metrics trees (`bShares.k1 - aShares.k1`, `_fmtSignedMult`, `_fmtSignedPp`, etc.). Reclaims ~46 lines of `_compute_deltas`/`_project_kind_minutes` porting. (`meta.a_start`/etc. carry the range labels CompareView's window-label header + length-mismatch banner need.)
- **D4 — preset day-math (mirror the Python helpers, LOCAL calendar).** WoW: A = [thisMonday−7, thisMonday−1], B = [thisMonday, thisMonday+6]. MoM: A = prev calendar month (full), B = this calendar month (full). today-vs-trailing: A = [today−7, today−1] (7-day baseline), B = [today, today] (single day). All anchored on `Local::now().date_naive()`, resolved to inclusive `NaiveDate` bounds Rust-side (the FE sends only the preset string). Custom: FE sends two explicit epoch-ms spans; each maps to the local days it touches (mirrors the existing `QueryWindow::Custom` → Range day-mapping).
- **D5 — color-family bar blocks (closes COLOR-FAMILIES SURFACE).** Two sites in EffectivenessRow:
  - **blocking-split bar:** `agent→human` share (AI made human wait = AI-family) → `colorForKind("ai-doing")`; `human→agent` share (human reviewing = human-family) → `colorForKind("reviewing")`. Replaces the source's `CT_TOKENS.active`/`textMuted`.
  - **concurrency-mix bar:** k=1/2/3/4+ strata are **engagement levels, not kinds** (the one site not strictly per-kind). Use a graded engagement ramp from `CT_TOKENS` (e.g. `surfaceAlt`/`ai-reasoning`/`ai-doing`/`subagent` as a low→high salience ladder, or a dedicated k-ramp) — build picks the exact tokens; the point is family-consistent + monotone-salient, NOT the source's raw `active`/`textPrimary`. Documented as a build micro-choice; not routed through `familyOf` (no kind to map).
- **D6 — RangePicker reuse.** The Custom preset's A/B date pickers reuse the existing `src/components/workspace/dashboard/RangePicker.tsx` (`onChange(startIso, endIso)`, `startIso`/`endIso`/`maxRangeDays` props) — no new picker. Two instances (A + B), each `onChange` updates the corresponding side and refetches.
- **D7 — FE pure-logic home.** Delta-formatting + share/ratio math (the `_fmtSignedMult`/`_fmtSignedPp`/`_fmtRelPct`/`_fmtSignedDurMs` family + `buildShares`) extract to a pure `compareMath.ts` (sibling to `metricsMath.ts`), unit-tested with vitest. `fmtMsDur`/`fmtMult` reused from `metricsMath.ts`.

## No 3rd-party probe needed
The Python↔Rust `build_comparison_data`/CompareView contract is the internal "3rd-party" boundary; the WP1 probe froze the segment-model contract and WP6c-research validated the metric re-derivation on the real dev DB. Producer is de-risked (reuses shipped `build_metrics` + `build_range` day-math). No external network API.

## Retrospect
- **What changed in our understanding:** The FE ↔ BE contract was smaller than the source implied. The archived WP6c spike had already found the `deltas` map dead (CompareView recomputes FE-side), so the Rust producer was `{a,b,meta}`-only. What the live render taught us — that the *source's own* blocking-split math (each component ÷ engaged-wallclock) silently overflows past 100% because both components overlap the merged wallclock — was only visible by driving real machine-global data, not from reading the source. The bar "looked fine" via flex-normalization; the bug lived in the tooltip %. That's the value of the live-bridge verify-self tier over a static wiring trace.
- **Assumptions that held:** The plan's D1–D7 were all correct as written — the `{kind:"compare", spec}` command shape mirrored WP6c-1's `Metrics` precedent cleanly, `build_metrics` reused per A/B side with zero changes, the RangePicker reuse worked, the color-family seam (`familyOf`/`colorForKind`) took the 2 bar blocks without modification, and dropping `_computeMetricsView` (D2) was right (no kind-filter seam exists to project through). Phase 1 froze the DTO and Phase 2 consumed it with no drift — the two-phase split held.
- **Assumptions that were wrong:** (a) The blocking-split bar math (ported faithfully from the source) was subtly wrong for a stacked "split" bar — fixed at verify-self by normalizing to fractions-of-sum. (b) The plan didn't anticipate that the split-normalization would make `topBlockingShift`'s two-branch logic dead (`haShift === −ahShift`) — caught by the reviewer, backlogged.
- **Approach delta:** Built almost exactly to plan. Two off-plan additions, both caught by the verify loop rather than the plan: the blocking-split normalization (verify-self live-render) and the codify-time overlapping-custom-windows test (a real gap the plan's test list didn't name). The producer taking union-rows + partitioning internally (vs the Python's pre-partitioned `events_by_day_a/_b`) was a cleaner API choice made at build, not plan.
