## Problem Statement

Closing a workspace whose CC is mid-work — or quitting the whole app while any
workspace is busy — destroys in-flight work with **no confirmation**. Today only the
QoL-WP1 *dirty-editor* guard prompts before a workspace close; a workspace whose CC is
`running`/`awaiting_input` (an active turn, or a prompt the operator hasn't answered)
is torn down silently, and the native macOS Quit (⌘Q / red window button) routes
straight to the backend `CloseRequested` handler which unconditionally `kill_all`s
every CC child — no prompt, no "3 workspaces are busy" warning. WP2 adds an
**active/busy** confirmation alongside the existing dirty-editor gate: closing an
active workspace prompts, and quitting the app while any workspace is active prompts
(and can enumerate which). It **extends the existing close-confirm machinery** in
`App.tsx` (`dirtyProbes` / `requestClose` / `resolveClose` + the shared `ConfirmModal`
+ the `confirmDialog.ts` spec factory) rather than building a parallel one, and reuses
the backend's already-broadcast per-workspace CC status (`useWorkspaceStatus` →
`stateFor`) as the "active" signal — no new backend state, no fragile
foreground-process detection.

**Design prior in play:** `[PRIOR: explicit-selectable-mode-over-inferred-mode]
leaning CC-status-based active-detection over fragile foreground-process detection for
v1 — flag if wrong.` The per-workspace CC status is already known and reliable; a raw
right-panel terminal's "running command" is NOT cheaply detectable (the shell is a raw
PTY with no foreground-process signal on the backend — confirmed at plan-time
code-seam check), so v1 scopes "active" to the CC status. See the Phase 1 spec-decision
task.

## Code-seam facts (verified at plan-time — grounds the tasks)

1. **Per-workspace close gate (Phase 1 seam):** `App.tsx` holds `dirtyProbes`
   (`useRef<Map<id, () => number>>`), `requestClose(id)` (reads the dirty probe → fires
   `ConfirmModal` via `setPendingClose` when `dirty > 0`, else `closeWorkspace(id)`),
   and `resolveClose(choice)`. The Filmstrip × and close-tab keyboard path both route
   through `onClose` → `requestClose`. The modal renders from a pure spec
   (`closeWorkspaceSpec(name, count)` in `confirmDialog.ts`), tested in
   `__tests__/closeWorkspaceGuard.test.ts`. **CC status is already available** at
   App-level: `const { stateFor } = useWorkspaceStatus(workspaces)`,
   `stateFor(id): "idle" | "running" | "awaiting_input" | "unknown"`.
   **"active" = `running` || `awaiting_input`.**
2. **App-quit path (Phase 2 seam):** the macOS Quit is a `PredefinedMenuItem` `.quit()`
   (`app_menu/mod.rs:248`) → the native window `CloseRequested` (single-window Tauri:
   the red button AND ⌘Q both fire it). `lib.rs:493` `on_window_event`'s
   `WindowEvent::CloseRequested { .. }` handler currently `kill_all`s **unconditionally**
   — it does NOT call `api.prevent_close()` and destructures `{ .. }` (drops the
   `api`). To confirm before quit, the handler must bind the `api`, `prevent_close()`
   when any workspace is active, and route a confirm request to the frontend; on
   "quit anyway" the frontend re-triggers the real quit. **No frontend quit-intercept
   exists today** — this round-trip is the one genuine new architecture in WP2.
3. **Right-panel terminal (Phase 3 seam):** the 2nd right-panel terminal is a raw
   `spawn_shell` PTY (`cc_session/mod.rs:329`, `resolve_shell_argv` = `$SHELL -l -i`).
   **No foreground-process / running-command signal exists** on the backend — detecting
   "a command is running" would need job-control introspection (fragile). Per the prior,
   v1 scopes terminal-active to CC-status-only (a raw-terminal workspace's *CC* status
   still governs); Phase 3 records that decision + a one-line note at the seam. (If the
   spec decides even a cheap heuristic is wanted, Phase 3 grows; default = document-and-scope.)

## Retrospect
- **What changed in our understanding:** The plan framed app-quit as "backend `CloseRequested` consults the aggregate before `kill_all`." The code-seam investigation at Phase 2 build revealed the native Quit is a `PredefinedMenuItem` `.quit()` → `CloseRequested` with **no frontend involvement today**, and that the FE already owns both the active-detection (`isActiveState`) AND the display names. That flipped the architecture to FRONTEND-DECIDES (backend just `prevent_close` + a bare signal), which is strictly cleaner — one source of truth for "active", zero backend duplication. Also learned: the MCP-bridge `ipc_emit_event` double-encodes its payload as a JSON *string*, so it can't drive the real `applyStatusUpdate` fold — the reliable way to seed a live status for verify-self is the React `setStatusMap` dispatch via the fiber (not an emitted event).
- **Assumptions that held:** `isActiveState` as the single shared predicate worked exactly as intended across both gates. The QoL-WP1 close gate extended cleanly (reasons-object `closeWorkspaceSpec`). The design prior (`explicit-selectable-mode-over-inferred-mode`) correctly steered the terminal-active scope to CC-status-only. The MCP-bridge live-verify approach (proven since M5) again carried a backend-lifecycle feature further than expected — it drove a REAL `quit_now`→clean exit-0.
- **Assumptions that were wrong:** (1) That the backend would decide the quit (see above — FE decides). (2) That I could hold a CC turn in `running` long enough to manually click × for verify-self — CC turns complete faster than the inter-tool latency, so the reliable driver is a seeded status, not a real held turn. (3) That an `invoke` interceptor could safely test the idle-quit path — `App.tsx` imports `invoke` from `@tauri-apps/api/core` at module load (its own ref), bypassing the `__TAURI_INTERNALS__.invoke` wrap, so the real `quit_now` ran and exited the app (which turned out to be *stronger* evidence).
- **Approach delta:** 3 phases as planned (per-workspace close → app-quit → terminal-scope note). Phase 2's internal architecture changed (frontend-decides) but the phase boundary + observable outcomes held. Phase 3 confirmed no code gap (documentation-only), exactly as the plan anticipated ("if Phase 1 decision-a covers it, Phase 3 is doc-only"). No back-loops, no redirects — clean autopilot run.

### Strengths

- The FRONTEND-DECIDES architecture is well-justified in-comment and avoids duplicating "what counts as active" in Rust; the backend's new surface is minimal (`prevent_close` + a bare event + one command).
- `perform_quit_teardown` extraction is a clean refactor: exact inline `CloseRequested` body lifted verbatim into a named fn with a single-teardown-path-invariance comment; both `quit_now` + `CloseRequested` route through it — no double-teardown, no behavior change.
- Defensive fallback on `emit("quit-requested")` failure (direct teardown + exit) prevents the "unquittable app" failure mode.
- The `quit-requested` listener reuses the proven latest-ref + `cancelled`-guard StrictMode pattern from the adjacent `menu` listener.
- Thorough pure-layer test coverage (`isActiveState` both branches, all four `closeWorkspaceSpec` reason-combos incl. no-reason, five `quitWhileActiveSpec` cases).
