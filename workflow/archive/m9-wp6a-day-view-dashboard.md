## Problem Statement

M9's retrospective analytics surface has all its plumbing shipped (WP2 hook→SQLite writer, WP2.5 native signals, WP3 reclassifier, WP4 `time_analytics_query` command + FE DTO seam, WP5 default-OFF tracking toggle) but **no UI**. WP6a is the shippable MVP: the **day-view** dashboard rendered natively and dark-themed, so the operator can finally *see* the per-project time breakdown Claudesk has been measuring — the retrospective counterpart to the real-time filmstrip/PiP/menu-bar status surfaces.

The source is `_ref/claude-customization/tools/claude-time/viz/dashboard.jsx` (4065 lines, 28 top-level components). WP1 proved the port is de-risked and split WP6 into 6a (day-view MVP, ~55% of the file) / 6b (week/month/side-panel) / 6c (metrics/compare, after WP3). **This spec covers 6a only.**

**[Updated 2026-07-08: surface corrected — GLOBAL view, not a per-workspace tab.]** The original plan placed the dashboard as a 4th tab **inside each workspace's `RightPanelHost`** (mounted per-workspace-instance). Operator correction at Phase-1 verify-human: **the dashboard is GLOBAL (all-projects) and must never belong to a specific workspace.** The data seam is already all-projects (`queryTimeAnalytics` is `scope: "global"`), so a per-workspace panel implied a workspace-scoping that does not exist. Root-problem *unchanged* (still "render the day-view analytics natively"); the *surface* changed. **New surface: a full-window GLOBAL view, mounted ONCE, that overlays/replaces the center stage** (the same shell pattern as `ProjectPicker`/`PickerOverlay` in `App.tsx`), toggled by a top-level control on the Filmstrip (beside the "+") **and** the `⌘⇧A` app-level chord (moved UP into `App.tsx` alongside `⌘⇧N`/`⌘⇧+digit`; removed from `panelHost.panelForChord`). Architectural principle (→ `arch.md` at WP7): a surface's mount-scope must match its data-scope. See `SURFACE-2026-07-08-M9-WP6A-DASHBOARD-IS-GLOBAL-NOT-PER-WORKSPACE`.

**Problem statement unchanged** (P3 F12 back-loops ×2, 2026-07-08) — root problem is still "render the day-view analytics natively"; the rejections were chrome refinements (drop the Most-used-tool stat; then two palette iterations for the human-family hue: warm→green after amber read as "warnings"), not a shifted root cause.

**Problem statement unchanged — F12 back-loop #4 confirmation (2026-07-08):** the 2 items that reopened Phase 3 are a **correctness bug** (overlap detection used the wrong definition of "overlap" — it must be same-project-only; cross-project concurrency is not an overlap) and a **UX access gap** (the global dashboard was gated behind opening a workspace; it must be reachable from the picker scene at launch). Neither shifts the root problem — both are refinements to *how* the day-view analytics surface behaves, not *what* it is. The overlap item is the sharper one: the operator corrected the definition three times, so the bar is exactly-right semantics + a codifying unit test, not a color tweak.

**Reuse from the discarded Phase-1 build:** `dashboard/DashboardPanel.tsx` (tracking-gate + empty-state + seed/listen), `dashboard/dashboardState.ts` (pure `dashboardMode`), `.dashboard-*` CSS, and the `React.lazy` scaffold all carry over — they move from the per-workspace slot to the new global-view host. **Discarded:** the `"dashboard"` `RightPanel` union entry, the panel-tab button + slot in `RightPanelHost`, and the `panelForChord` `⌘⇧A→panel` mapping.

## User Stories

- As the operator, I want a **Dashboard tab** in the workspace right-panel (alongside Editor / Diff / Terminal) that I open with a click or a ⌘⇧-chord, so I can review where my day's attention actually went across projects **without leaving Claudesk** (no CDN React, no separate browser window, no stale-tab-cursor problem).
- As the operator, I want the tab to show **today's per-project timeline** — session rows with tiled segments colored by kind (AI-doing / subagent / AI-reasoning / typing / reviewing / away), an hour ruler, a summary strip, and a legend — reading the same **dark, IDE-native palette** the rest of Claudesk uses.
- As a friend-user (or the operator on a fresh install) who has **tracking OFF**, I want the tab to show a clear **"enable tracking to see analytics" empty-state** instead of a broken/blank view — and to update live the moment I flip the WP5 toggle on.
- As the operator, I want the dashboard chunk **lazy-loaded** so it doesn't inflate app startup parse time (it's a heavy chunk that only matters on first open) — folding in `SURFACE-2026-06-19-CM6-BUNDLE-SIZE-LAZY-LOAD`.

## Acceptance Criteria

The feature is done when:

1. **Global-view wiring.** The dashboard is a **top-level view**, NOT a right-panel tab. `App.tsx` gains a `showDashboard` state; a top-level control on the `Filmstrip` (an analytics button beside the "+", `data-testid="filmstrip-open-dashboard"`) and the `⌘⇧A` app-level chord (capture-phase document listener in `App.tsx`, pure `isDashboardChord` predicate) both toggle it. When on, a single `<GlobalDashboard>` overlays/replaces the center stage (the `PickerOverlay` render-gate pattern); a close/back affordance dismisses it. It mounts ONCE (not per-workspace); workspaces stay mounted underneath. A `role="tab"`/panel wiring in `RightPanelHost` is NOT added (and any Phase-1 per-workspace wiring is removed).
2. **Day-view render (tracking ON, data present).** Opening the dashboard renders — in the **dark palette** — the day-view chain: `Toolbar` (Day label; Week/Month/Custom/Compare controls present-but-inert in 6a, see Out of Scope), `SummaryStrip`, `Legend`, `HourRuler` + `HourGridBackground`, per-project `ProjectHeaderRow`s, `SessionRow`s with tiled `SegmentBar`s in the WP6 semantic colors, `OverlapMarkerLayer`, and `CollapsedTrackRow` (collapse/expand). Data comes from `queryTimeAnalytics({ kind: "day" })` (WP4, `scope: "global"` — all projects), **not** `window.CT_DATA`.
3. **Kind→color mapping (6-kind).** All segment/legend/summary rendering uses the **shipped WP4 6-kind kebab enum** (`ai-doing` / `subagent` / `ai-reasoning` / `typing` / `reviewing` / `away`), NOT the source's 5-kind (`active`/`reading`/`thinking`/`subagent`/`away`). The palette encodes the AI-vs-human **color-family** split (`SURFACE-2026-07-06-M9-COLOR-FAMILIES-AI-VS-HUMAN`): AI-execution family (`ai-doing`/`subagent`/`ai-reasoning`) in one hue family, human family (`typing`/`reviewing`/`away`) in another — starting from WP1's locked darker-fill deep tokens + the `textOn(bg)` luminance-ink helper (lifted verbatim). Exact hue assignment settled at plan/build (see Open Questions).
4. **Empty-state (tracking OFF).** When `getTimeTrackingEnabled()` → `false`, the dashboard view still opens but shows an "enable tracking to see analytics" empty-state (not the day breakdown). It flips live to the day view (and back) when the `TIME_TRACKING_ENABLED_EVENT` broadcast fires — no remount, no manual refresh. When tracking is ON but the day simply has no rows yet, a distinct "no activity recorded today" empty message renders (not the same as the toggle-OFF message).
5. **Lazy-load.** The dashboard component is loaded via `React.lazy` + `Suspense` (with a lightweight fallback), so its chunk is not in the app's initial bundle. The EditorPanel/CM6 lazy-load is folded in on the same axis. `pnpm build` no longer prints the 500 KB chunk-size warning for the main bundle (or the main entry chunk measurably shrinks — captured in verify-auto).
6. **Gates green.** `tsc --noEmit` clean (the ported components are real TSX, not `@ts-nocheck`), `eslint` clean, `pnpm vite build` exit 0, Rust unchanged (no backend change in 6a — WP4's command already exists), all existing FE + Rust tests still pass, new tests cover the pure logic added (chord/panel wiring, kind→color map, empty-state predicate).
7. **Live verify-self.** Via the MCP bridge + a scratch workspace: enable tracking, drive a CC turn, open the global dashboard (button + ⌘⇧A), confirm a dark per-project day breakdown renders; toggle OFF → the empty-state shows; toggle ON → the breakdown returns; the dashboard is the same regardless of which workspace is focused (global).

- **Metrics / Headline / Compare / Effectiveness / PresetSelector panels** → WP6c (they consume `metrics`/`comparison` shapes not produced by WP4's day query, and depend on WP3 metric defs already locked but not surfaced by WP4 yet).
- **Range navigation / date picker / prev-next-day.** 6a shows **today** only. Day-switching is WP6b's range-nav.
- **`ProjectFilterPopover` interactive filtering.** The legend renders; per-project show/hide filtering can be deferred to 6b unless it falls out cheaply (decide at plan).
- **Any backend/Rust change.** WP4 shipped the `time_analytics_query` command + DTOs; WP5 shipped `time_get_tracking_enabled` + the broadcast. 6a is a pure-frontend consumer of both. (If a gap is found in WP4's day payload during the port, that's a back-loop / SURFACE, not in-scope new backend work.)

## Technical Constraints

- **The real data seam is WP4's 6-kind kebab DTO, not WP1's baseline.** `src/state/timeAnalytics.ts` is the frozen FE mirror: `queryTimeAnalytics(window, scope)` → `TimeAnalyticsResult` = `({kind:"range"} & RangePayload) | ({kind:"week"} & WeekPayload)`. 6a calls `{ kind: "day" }` → a **`range`** result (1-day: `iso` + `hour_range` present, `projects: ProjectPayload[]`, each with `sessions: SessionPayload[]`, each `segs: SegPayload[]` where `SegPayload.kind` is the 6-kind kebab tag; `SegPayload.label` present only on `subagent`). **snake_case field names are verbatim** (the IPC-DTO casing convention — Tauri does NOT camelCase-convert). This *differs* from the source `dashboard.jsx`, which reads `window.CT_DATA.day_payloads_by_iso[<iso>]` with the OLD 5-kind enum — so the port is **structure-mechanical + a kind-enum remap + a data-source swap**, not the pure verbatim lift WP1's spike did (the spike proved *rendering feasibility* against the old fixture; the real port adapts to the new seam). This is the single largest bounded delta from WP1's port recipe and must be threaded through every `kind ===`/`FILTER_KINDS`/`byKind`/`renderOrder`/`segStyle`/legend/`sumActive`/`sumKind` site (~15, enumerated in the plan).
- **Port recipe (WP1, §b).** `import React from 'react'` fixes all namespace-form hook call-sites (`React.useState` etc.) with zero body edits; the component is self-contained (no external chart dep — hand-rolled SVG). Lift WP1's dark `CT_TOKENS` block + `textOn()` verbatim, then recolor per the 6-kind family map. WP1 spike source lives (gitignored) at `tmp/dark-render-spike/dashboard.spike.tsx` as the token reference.
- **Chord choice — `⌘⇧A`, now an APP-LEVEL chord.** The taken ⌘⇧ chords are **E** (editor), **D** (diff), **T** (terminal), **O** (freed), **P** (palette), **F** (project-search); ⌘⇧+digit is workspace-switch. `⌘⇧A` (Analytics) is free and mnemonic. **[Revised — global surface]** It is registered in **`App.tsx`** as an app-level capture-phase document chord (alongside `⌘⇧N` new-workspace + `⌘⇧+digit` workspace-switch), toggling the global dashboard view — NOT in `panelHost.panelForChord`. A pure `dashboardChord.ts` predicate (`isDashboardChord(e)`) is the vitest-testable core, mirroring `newWorkspaceChord.ts`/`workspaceSwitchChord.ts`.
- **Global-view surface (CLAUDE.md "one window, many workspaces").** The dashboard is a **top-level view**, mounted ONCE, overlaying/replacing the center stage — the `PickerOverlay` pattern (`App.tsx` `showPicker`). It does NOT add an OS window (single-window rule intact) and it is NOT per-workspace. Workspaces stay MOUNTED underneath (the overlay hides the stage via render-gating, same as the picker overlay). Lazy-mounted via `React.lazy` + `Suspense` (chunk loads on first open).
- **Dark-mode only (CLAUDE.md).** No `prefers-color-scheme: light`, no light tokens. The dark `CT_TOKENS` are the only palette.
- **No new dependency.** The port adds no npm package (hand-rolled SVG, single `import React`). No new Rust crate.
- **Verify-self posture.** Frontend/UI feature → the agent drives live verify-self through the `tauri` MCP bridge against a **scratch workspace** (per CLAUDE.md M5-WP2 + M9-WP2 caveats), NOT `feature-verify-self-runner`. Bridge teardown per caveat (d): driver stop + kill dev by exact PID + `lsof -ti tcp:1420 tcp:9223 | xargs -r kill -9`; pre-clean ports before `pnpm tauri:dev`.
- **`[PRIOR: new-surface-must-earn-its-place-against-existing-ones]`** — the global Dashboard does NOT overlap the real-time status surfaces (filmstrip/PiP/menu-bar): those are *live glance*, this is *retrospective analysis*. Genuine non-overlap → prior satisfied, no shrink needed. Noted, not a scope-changing firing.

Rationale: **toolbar geometry is final now**, so 6b/6c only *wire* the controls rather than re-laying-out the toolbar twice. `[PRIOR: explicit-selectable-mode-over-inferred-mode]` agrees (a legible, directly-visible set of view modes) — take it.

## Resolved Decisions (plan, 2026-07-08)

- **`ProjectFilterPopover` → DEFER interactive filtering to WP6b (RESOLVED at plan).** The source's `ProjectFilterPopover` renders a trigger + a checkbox popover backed by `FilterContext` (in the standalone it's a no-op provider). Wiring *real* per-project show/hide filter state adds a stateful interaction surface (context provider + filter application threaded through `DayTimeline`/`SummaryStrip`/overlap detection) that isn't part of the day-view MVP's value. **6a renders the static `Legend` only** (the kind→color key) and OMITS the filter popover trigger; the `FilterContext` default (all-visible) is used so `DayTimeline` needs no filter plumbing. WP6b adds the interactive popover + range nav together. `[PRIOR: new-surface-must-earn-its-place-against-existing-ones]`'s spirit (scope to the irreducible core) agrees.
