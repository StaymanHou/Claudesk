# Feature: M7 — Menu-bar status item (ambient alarm + actuator)

## Problem Statement

When the Claudesk window is hidden, minimized, full-screen, or on another macOS Space, the operator has no zero-effort signal that a project is **blocked on their input**. The M4 filmstrip is in-window only; the M5 PiP covers the out-of-window case but is a panel you summon/pin and place — a *deliberate, detailed* look. What's missing is a **passive, ambient, zero-allocation** signal in a place the operator's eyes already pass over all day: the macOS menu bar.

M7 adds a **menu-bar (system-tray) item scoped to exactly the one thing the menu bar can do that the shipped PiP cannot** — exploit its *location*. It is **not** a third status surface (a popover dashboard would be a strict subset of PiP — cut as redundant). It is **(a) a 2-state ambient alarm** ("is any project waiting on me?" — lit when any workspace is `AwaitingInput`, neutral otherwise) **+ (b) a native actuator menu** (Show Claudesk / Toggle PiP / Quit — acts on the app, which display-only PiP can't). It subscribes to the existing M3 `status_broadcaster` `workspace-status` event (no broadcaster change).

## User Stories

- As the operator running 3–4 CC sessions across rotating projects, I want a **menu-bar glyph that lights up when any project is waiting on my input** so that — even with the Claudesk window hidden, full-screen, or on another Space — I get a zero-effort ambient "stop, a project needs me" signal in a strip I already watch all day, without summoning or placing any panel.
- As the operator, I want **clicking the menu-bar item** to offer Show Claudesk Window / Toggle PiP / Quit, so the common app-level actuators are reachable from the menu bar even when the window is hidden.

## Acceptance Criteria

The feature is done when:

1. **A menu-bar (tray) icon is present** whenever Claudesk is running, as a template image (adapts to light/dark menu bar).
2. **The icon is LIT when ANY open workspace is `AwaitingInput`, and NEUTRAL otherwise** (Running and Idle both read as neutral; zero workspaces → neutral). It updates live on every `workspace-status` event and on workspace register/deregister.
3. **The lit/neutral reduction is a pure, unit-tested fold** (`aggregate_alarm(states) -> {Attention|Neutral}`), independent of a live app.
4. **Clicking the icon opens a native menu** with three actuators: **Show Claudesk Window** (brings the main window forward) / **Toggle PiP** (flips the existing `pip-mode`) / **Quit** — each performing its existing app action.
5. **The alarm is visible when the Claudesk window is NOT in focus** — on another Space, minimized, or behind a full-screen app (the milestone's point). Verified on the installed `.app` (operator-carry / DEFERRED-TO-RELEASE for the native-glyph + out-of-focus checks; the agent GREENs the bridge-observable slice).
6. Dev/prod isolation holds: one tray icon per running identity (`com.claudesk.app` vs `.dev` don't collide).

## Out of Scope (and explicitly CUT)

- **A popover `WebviewWindow` / per-workspace status list / navigate-on-click** — **CUT** at the spec debate (2026-06-29). A popover dashboard would be a strict subset of the shipped M5 PiP (live mirrors + layouts + attention weighting + all-Spaces + near-zero-pixel `minimal`/`On`). "Which project / jump to it" is a *detail* interaction → PiP's job or the main window's. (design-prior `new-surface-must-earn-its-place-against-existing-ones`.)
- **`tauri-plugin-positioner`, a third Vite entry (`popover.html`), `src/popover/`** — all CUT with the popover. No new dependency, no new webview.
- **A 3-state dot (green/blue/amber) / running indication in the menu bar** — CUT. The menu bar carries one bit (waiting-on-me / not); running-vs-idle detail lives in PiP / the window.
- **The workflow-doc-hierarchy watcher / workflow-position line** — re-anchored to M8 (no popover to host it; `SURFACE-2026-06-22-WP5-DROPPED-WATCH-WORKFLOW-DOC-HIERARCHY`).
- **Changes to the M3 `status_broadcaster`, PiP behavior, or the macOS activation policy** — M7 only subscribes; the tray is additive (dock icon + main window kept, `ActivationPolicy::Regular`).
- **Real accelerators on the menu items** — mirror existing actions only (the 2026-06-24 app-menu discipline; accelerators steal webview keystrokes).

## Technical Constraints

- **Architecture is designed (shrunk):** `arch.md` §B.2 (DESIGNED-FOR-BUILD — SHRUNK, Revision 2026-06-29) + the Key Decision "Menu-bar item is an ambient ALARM + ACTUATOR, not a status surface." Build to that.
- **Tauri 2.11.x API facts verified at the arch pass** (no 3rd-party probe gap): `TrayIconBuilder` (core Tauri); `icon_as_template(true)`; `set_icon_and_icon_as_template_atomic(icon, true)` to swap lit↔neutral WITHOUT the `tauri#6527` template-flag-reset blink; native menu via `TrayIconBuilder::menu(...)` (default click-to-show — we do NOT call `show_menu_on_left_click(false)` since there's no popover to free the left click for).
- **New module** `src-tauri/src/tray/` (mirrors the `pip/` shape: pure `aggregate_alarm` fold + DTO in `mod.rs`, tray-icon ops + commands in `commands.rs`).
- **New `trayIcon` block** in `tauri.conf.json` + 2 pre-rendered template PNGs (lit + neutral) in `src-tauri/icons/tray/` (no runtime image generation; load as `tauri::image::Image`).
- **Reuses (no new seam):** the M3 `workspace-status` broadcast + snake_case DTO; the M5 **run-on-main-thread rule** (tray-icon ops are AppKit ops — any background-thread caller marshals via `app.run_on_main_thread(...)`); the 2026-06-24 `app_menu` `on_menu_event`/emit + managed-handle bridge (the menu mirrors existing actions); a thin bring-forward command for "Show Claudesk."
- **Verify-self / MCP bridge:** the `aggregate_alarm` fold + DTO are unit-tested (the bulk of the logic). The icon-swap path is exercised by driving a status transition on a scratch workspace (`tmp/scratch/scratch-a`) and confirming the app stays alive (no abort). The **native menu-bar glyph (lit/neutral)** + the **native menu** are NOT in any webview → **operator-carry / DEFERRED-TO-RELEASE** (M5/M6 native-surface pattern). After any bridge session: `lsof -ti tcp:1420 tcp:9223 | xargs -r kill -9` + `driver_session{stop}` (the M6-WP7 port-cleanup caveat).
