## Problem Statement

M10's updater engine is fully built (WP2 core flow, WP3 brew detect-and-defer, WP4
user-control UX, WP5 `/release` pipeline — all shipped, uncommitted-to-remote on the
batched M9+M10 tree). Three load-bearing behaviors were **legitimately deferred to
WP6** because they only fire on a REAL published release + a REAL installed build:
WP1's unsigned-relaunch Gatekeeper verdict (GO self-clear vs FALLBACK instruct-user),
WP3's real-`/Caskroom/` brew detection, and WP5's real live publish + self-update.
WP6 is the milestone-exit gate that verifies all of them end-to-end. It also folds in
the one substantive WP4 code gap: the `useUpdater` error surface (`phase==="error"` /
`errorMessage`) is produced but has **zero consumers** — an apply failure surfaces
nothing (the banner silently reverts), a real hole in the "user is always in control"
promise. That failure path first fires live at WP6, so it's wired + verified here.
The FLOOR insight: the current M10 tree (v0.2.5, updater-capable) IS the first
updatable build, so WP6's "older build to update from" is the v0.2.5 floor itself —
no separate old version is manufactured; publish v0.2.6 and verify the floor
self-updates to it.

**Prior art de-risking the 3rd-party surfaces:** WP1 (the completed probe WP) already
verified the `tauri-plugin-updater` artifact/manifest/minisign shapes empirically; the
engine is built through WP2–WP5. No new probe gate. Preserved WP1 artifacts + probe
keys at `~/claudesk-m10-probe/`; the FRESH release key (`774E2E8429FDF78A`) at
`~/.claudesk-release-keys/` is the one WP6's live release MUST use (NOT the probe key).

**[Revised 2026-07-17 — F23 back-loop: reprioritized to brew + found a brew-check gap.]**
The original plan made **direct-download self-update (C1) the load-bearing check** — this
CONTRADICTED the operator's standing directive (recorded at WP1 verify-human,
`SURFACE-2026-07-17-M10-WP1-LIVE-VERIFY-DEFERRED`): **brew is the priority; direct-download
is lower-stakes** (the operator + friends install via brew, which never self-installs, so
the self-clear/Gatekeeper machinery never runs for real users). Two corrections: **(1)**
direct-download self-update verification (incl. the WP1 GO/FALLBACK Gatekeeper verdict) is
**DEMOTED to a standalone WP after this one** — removed from WP6's live pass, stays deferred.
**(2)** a real **behavioral gap** surfaced: `updater_check` currently **short-circuits
BEFORE the network `check()`** for Homebrew (`commands.rs` L90) → a brew user is told "run
`brew upgrade`" **even when up to date**, and never learns whether an update actually exists.
Operator's target: a brew install must do a **REAL check** and get the **same
update-notification experience** as direct-download — only the *action* differs (the Update
button becomes a **one-click-to-copy `brew upgrade claudesk`** instruction). This is a NEW
CODE phase (Phase 2), not just verification. Phase 1 (the error-surface fold) already
shipped and is unchanged. WP6 = Phase 1 (done) + Phase 2 (brew real-check-and-notify code)
+ Phase 3 (brew-focused live milestone-exit pass).
