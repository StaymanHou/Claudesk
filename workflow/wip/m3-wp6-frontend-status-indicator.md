## Problem Statement

M3 built the backend "central nervous system" end-to-end in automated tests — WP2 installs the hook, WP3 binds the `AF_UNIX` listener and parses typed `HookEvent`s on a dedicated thread, WP4 transforms each event into a snake_case `WorkspaceStatusUpdate` and emits it on the `workspace-status` Tauri event — but **no frontend consumes it, no workspace is ever registered in the `WorkspaceRegistry`, and the full chain has never run live against a real `claude`.** WP6 closes the loop: the React webview subscribes to `workspace-status`, renders an honest idle/running/awaiting-input/unknown indicator on the workspace chrome, and wires workspace-open → `WorkspaceRegistry::register` / close → `deregister` so WP4's cwd→workspace matching finally has data to match against. This is the M3 verify surface — a real CC state transition observed in the UI purely from the hook channel (never PTY scraping) — and it carries the three deferred live confirmations (WP1 `Notification` payload, WP3 live socket bind, WP4 live emit) that the M3 plan parked here. On completion the 3 item-scoped `#[allow(dead_code)]` WP4 left (naming WP6 as removal owner: `WorkspaceState::Unknown`, `WorkspaceRegistry::register`, `WorkspaceRegistry::deregister`) become live and are deleted.

## Key Context (load-bearing facts for build)

- **Wire contract is fixed by WP4 and must be mirrored verbatim.** Event name = `"workspace-status"` (Rust const `status_broadcaster::commands::STATUS_EVENT`). Payload = `WorkspaceStatusUpdate`, **snake_case end-to-end** (arch.md convention, pinned by `dto_serde_shape_is_snake_case`): `workspace_id: string`, `state: "idle"|"running"|"awaiting_input"|"unknown"`, optional `last_event_at?: number` (epoch ms, omitted when absent), optional `last_output_snippet?: string` (omitted when absent). The TS interface MUST use these exact keys + the snake `awaiting_input` state literal — no camelCase, no kebab. This is the M2 IPC-DTO lesson now codified.
- **Existing kebab `WorkspaceStatus` is a DIFFERENT type — do not conflate.** `src/state/workspace.ts` already has `WorkspaceStatus = "idle"|"running"|"awaiting-input"|"unknown"` (kebab `awaiting-input`) on the `Workspace` model, currently a static `"idle"` placeholder set by `makeWorkspace`. The IPC wire type is its own type (snake `awaiting_input`). Plan keeps the wire type separate (new module) and updates the live workspace status from incoming events; the presentational label/dot derives from the live state. Do NOT retype the wire shape to kebab — the boundary mirrors the backend verbatim.
- **Registration seam is already built + tested in WP4.** `WorkspaceRegistry::register(&Path, String)` / `deregister(&Path)` exist behind `SharedRegistry = Mutex<WorkspaceRegistry>`, already `app.manage(init_registry())` in `lib.rs` `.setup()`, already read per-event by the drain thread. WP6 adds only: (a) two `#[tauri::command]`s that lock the managed `SharedRegistry` and call register/deregister, (b) their registration in `generate_handler!`, (c) frontend `invoke` on open/close. The `workspace_id` registered must equal the id the frontend keys the indicator by (the `ws.id` from `state/workspace.ts`).
- **`register` takes the PROJECT PATH, not the cwd string.** It canonicalizes internally; `resolve_cwd` canonicalizes the incoming hook `cwd` the same way. So registering `workspace.project_path` makes a hook fired from that project dir (CC's `cwd`) resolve. The IPC command takes `project_path: String` + `workspace_id: String` (mirror the `sublime_open` arg shape: snake field → camel `projectPath`/`workspaceId` on the TS `invoke` call, Tauri's default arg-key convention — confirmed by `cc_input { sessionId }` usage).
- **IPC errors must never be swallowed.** The register/deregister `invoke` calls follow the WP6-picker / WP7-M2 lesson: a rejection is surfaced (at minimum `.catch` logging to console; a missing registration silently breaks status, so it must not vanish). Register is best-effort-but-loud; the indicator falling back to Unknown is the honest visible failure mode.
- **Frontend test posture (WP5/WP7 precedent).** Pure logic → vitest (no jsdom/RTL): the status reducer/mapping (`WorkspaceStatusUpdate` → indicator presentation) is the unit-test target. Live DOM/PTY/IPC → verify-self/human against the real app. The `listen`/`invoke` wiring itself is NOT unit-tested (it's the AppHandle/runtime-bound plumbing, same design property as XtermPane's listeners + the WP4 emit line).
- **Native-PTY verify discipline.** WP6's live check is NOT browser-harness-observable — CC runs in a native PTY inside the Tauri webview, and the hook→socket→emit chain only exists under the real `pnpm tauri dev` app. Verify via the real app + a real `claude` (observe `ps`/screencapture/the visible indicator), per the `verify-native-pty-via-ps-screencapture-stderr` posture. Playwright cannot drive this.
- **Build hygiene:** `lsof -ti:1420 | xargs kill` before `pnpm tauri dev` (strictPort). Kill the dev app before `cargo test` (shared `target/` lock). `cargo test` ~5s (registry timeout 120000ms); full suite was 180 tests at WP4. `pnpm test` (vitest) ~1s.
