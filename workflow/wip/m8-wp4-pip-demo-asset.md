## Problem Statement
M8's second value-showcase asset: a short looping GIF that *shows* the picture-in-picture value prop — Claudesk's always-on-top PiP panel, pinned in a screen corner over a faux "another app" (editor / video) backdrop, keeps a CC session **monitorable while the operator works elsewhere**, and **pings them the moment CC needs input** (a status dot flips to AwaitingInput). Narrative beat: *"Do your other work — CC stays watchable in the corner and pings you the moment it needs you."* Built on the existing `tooling/demo/` WP2 harness (reuses `shell.html`'s `data-region="pip"` view, the real status-dot CSS via `_dots.generated.css`, and the WP3 layers `busyAt.js` / `frameAt.js`). Output: `npm run pip` → `out/pip.gif`, under the 3MB budget. This is a marketing/communication asset (README + GitHub release + M13 launch), so its acceptance is operator visual-judgment at the 4.4 checkpoint, not an automated assertion of "looks good."

**Problem statement re-check (F12 round-1 re-entry, 2026-06-29):** Unchanged — the deliverable is still a looping GIF conveying the PiP monitor-while-elsewhere value. Round-1 learned the *execution* missed: a static corner status-list didn't read as "active" or as a live workspace *mirror*. The operator's direction (vertical active-mirror w/ live CC cadence, an active Slack backdrop, a fuller typing→ping→⌘+Tab→answer beat sheet, 2 workspaces) sharpens the same goal, not a new one. Re-authoring execution, not re-planning scope.

## Notes (plan-time context, from the WP2/WP3 harness study)
- **The shell already has a full PiP path.** `shell.html` has the `.backdrop` + `.pip` regions; `shell.js` has `renderPip(k)`, `T.backdrop`, `T.pipPos` (corner-pin), and region toggling via `body[data-region="pip"]`. `shell.css` has `.pip` / `.pip-row` / `.backdrop` chrome (lines 133–183, 274–283). So WP4 is mostly **authoring a timeline** + a small `renderPip` busy-meta extension — not new infrastructure.
- **The flagged WP4 decision (from the pause note) is resolved here:** keep the PiP as the **compact per-workspace status-row list** (`.pip-row`) — that IS the real PiP surface — and make it feel live by ticking the focused row's `meta` via `busyAt`, rather than embedding the full busy-stream stage inside the panel. The full code-cascade lives in the filmstrip demo (WP3); the PiP demo's job is the *ambient-monitor + ping* read, which a status list + a dot flip conveys more honestly than a shrunk CC terminal.
- **PiP is display-only in v1** (vision + CLAUDE.md): clicking a PiP tile does NOT bring the workspace forward. So — unlike the filmstrip demo — there is **no cursor-click-to-promote beat**. The ping (dot→awaiting blue blink) is the payload; the operator's takeaway is "I'll glance over / go back to Claudesk", not "I click the PiP".
- **Harness conventions to carry (WP2/WP3):** JS loads as classic `<script>` (NOT `type=module` — Chromium CORS-blocks ES-module `file://` imports on the capture path); timeline string fields inject via `innerHTML` (author-controlled dev input only); new pure helpers follow `frameAt`/`cursorAt`/`busyAt` (classic script → `globalThis.__x`, unit-tested in a sibling `.nodetest.mjs`). One-time setup already done on this host (`npm install`, chromium cached).
- **Project cast must stay UNRELATED projects** (philosophy: parallelism across PROJECTS, not agents-in-one-system) — reuse the WP3 cast vocabulary (catan-companion / tax-cruncher / hugo-blog / recipe-box) or equivalents; do NOT use one-system service names (the WP2 smoke timeline's anti-pattern, flagged for WP5 recast).
- **Verify posture:** asset is visual + subjective. Agent CAN drive verify-self statically (the `npm run pip` build IS an end-to-end run — Playwright captures real frames; assert legible looping GIF under budget, `npm test` green). The "does it convey the value" judgment is the operator's at verify-human 4.4 (genuine checkpoint — PAUSES even in autopilot, like WP3's did).
- **WP3 quality findings to optionally fold in** (`workflow/backlog-quality-findings.md` → `# m8-wp3-filmstrip-demo`): all 3 are MINOR/low-priority on dev-only tooling; not blocking WP4. The WP2 smoke-timeline naming recast is explicitly deferred to WP5.
