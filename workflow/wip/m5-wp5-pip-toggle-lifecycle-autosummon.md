## Problem Statement

The PiP NSPanel exists (WP1/WP3) with four layouts + a persisted switcher (WP4), but it has no lifecycle: there is no in-Claudesk control to show/hide it, it does not yet track the live workspace roster (open/close), and on app quit the teardown path has a known gap. WP5 is the lifecycle work package.

A scope decision settled this session (2026-06-27, operator) **adopts auto-summon / auto-dismiss** for the panel — a reversal of the standing "NOT auto-summoned" vision stance, made safe by being an **opt-in setting**. The motivating insight: the PiP's whole job (vision Success Metric 6 — "find the awaiting-input CC when Claudesk is NOT in focus, <1s, zero clicks") is best served by a panel that *appears on its own* exactly when the user has left Claudesk, rather than one they must remember to summon *before* tabbing away. The WP4 mirror-freeze fix (a backgrounded PiP now updates live instead of freezing) makes auto-summon genuinely useful rather than surfacing a stale panel.

**[F9b re-entry 2026-06-27] Problem statement unchanged** — verify-self surfaced an implementation defect (the auto-summon debounce showed the panel from a background thread → off-main-thread AppKit abort), NOT a shift in the root problem. Auto-summon is still wanted + feasible; only the show call's threading was wrong (now marshaled to the main thread). Fixing the symptom IS fixing the root cause here.

**[F12 re-entry 2026-06-27] Problem statement unchanged at root; regime-control MODEL revised** — verify-human exposed that the *inferred* regime (origin=Manual/Auto + manual_off) had a dead-end (no return to auto without relaunch). The root goal (auto-summon when away; user can pin or disable) is unchanged; what changed is HOW the regime is controlled: an explicit tri-state `PipMode` Off/On/Auto replaces the inferred origin/manual_off/pip_auto_summon model. Not a symptom-patch — a model correction that removes the unreachable state by construction.

The use-case analysis that drove the decision (operator scenarios):

| # | Scenario | Want auto-summon? | Why |
|---|---|---|---|
| 1 | Focus stolen by CC (Playwright/tauri-dev pops up); user snaps right back | **No** | <1s involuntary blur; a flash is pure noise, may cover the very window CC opened |
| 2 | ⌘Tab to browser to check a result, ⌘Tab back | **No** | Brief, purposeful; not monitoring CC; panel would overlay what you tabbed to see |
| 3 | Done with all CC, ⌘Tab / switch Space to watch YouTube while monitoring | **Yes (canonical)** | Sustained intentional departure — exactly what the PiP is for (Metric 6) |
| 4 | Telegram interrupt; ⌘Tab / switch Space to reply | **Yes** | Sustained departure; a corner glance surface is passively beneficial |

The discriminator between summon (3,4) and don't-summon (1,2) is **not the trigger event** (all four are "Claudesk lost focus") but **how long the user stays away** — hence a **debounce** on the blur trigger. Dismiss-on-refocus is wanted unanimously wherever a panel was summoned.

## User Stories

- As the operator, I want an in-Claudesk control to show/hide the PiP panel, so I can summon it manually without going through the (not-yet-built) menu-bar surface.
- As the operator, when I leave Claudesk for a sustained stretch (switch Space / ⌘Tab away to a long task), I want the PiP to appear on its own so I can monitor CC status without remembering to summon it first.
- As the operator, when I come back to Claudesk, I want the auto-summoned PiP to get out of the way on its own.
- As the operator, when CC briefly steals focus or I quick-check something and come right back, I do NOT want the PiP to flash up — only a sustained absence should summon it.
- As the operator, when I have explicitly toggled the PiP OFF, I want that respected — leaving Claudesk should not resurrect it.
- As a multi-monitor friend-user (a different setup than the operator's), I want to turn auto-summon OFF, because blur-on-the-other-monitor would make it misfire — and I still get the manual toggle.
- As the operator, when I open or close a workspace, I want the PiP roster to add/drop that workspace's tile to match.
- As the operator, when I quit Claudesk, I want the PiP panel to tear down cleanly with no orphan NSPanel.

## Acceptance Criteria

The feature is done when:

### Strand 1 — Manual toggle + lifecycle (original WP5 core)
- [ ] An in-Claudesk control toggles the PiP show/hide. Placement: a `View`-menu item in the existing native app menu (reuse the `app_menu` id→event→action seam, no accelerator) and/or a window-chrome button. (The menu-bar right-click "Toggle PiP" is an **M7** surface — note the forward-coupling, do NOT build it here.)
- [ ] The PiP reflects the live workspace roster: opening a workspace adds its PiP tile; closing one drops its tile. True across all 4 layouts. (`close_workspace` already deregisters from the broadcaster — confirm the PiP tile drops with it; subscribe to the same registry signal the filmstrip uses.)
- [ ] Clean teardown on app quit (`WindowEvent`/quit path): the panel closes via `to_window()`→`close()`; no orphan NSPanel leaks. Toggling off fully hides it AND stops PiP-specific render cost (the mirror-layout serialize loop), mirroring the filmstrip-collapse discipline.

### Strand 2 — Auto-summon / dismiss state machine (new)
- [ ] When the auto-summon setting is ON: the PiP **auto-summons after the main Claudesk window has been continuously unfocused for ~3s** (debounce; value tunable). A refocus *during* the debounce window **cancels** the pending summon — no panel appears (covers scenarios 1, 2, and rapid ⌘Tab oscillation).
- [ ] When auto-summoned, the PiP **auto-dismisses immediately on main-window refocus** (no debounce on dismiss).
- [ ] **Manual-off suppresses auto:** if the user has explicitly toggled the PiP OFF, a subsequent blur does NOT auto-summon it (explicit user action wins). The suppression clears on the next explicit toggle-on (and on app relaunch — see persistence below).
- [ ] **AwaitingInput is NOT a summon trigger.** A workspace entering AwaitingInput while Claudesk is focused does not summon the panel. AwaitingInput remains an *in-panel emphasis* signal only (the existing WP4 `orderForAttention` blink/glow / minimal-layout POP), surfaced once a panel is already shown.
- [ ] Auto-summon and the manual toggle coexist: a manually-summoned panel stays up on refocus only if... *(decision: auto-dismiss applies only to auto-summoned panels; a manually-summoned panel is dismissed by manual toggle, not by refocus — see Open Questions Q1 to confirm the manual-summon + refocus interaction)*.

### Strand 3 — Setting + persistence (new; extends the WP4 settings store)
- [ ] A new app-global setting `pip_auto_summon: Option<bool>` is added to `AppSettings` (`config_store/settings.rs`), **defaulting ON** (`None`/unset reads as `true`). Forward-stable optional field per the store's existing convention.
- [ ] A Settings UI surface (or, minimally, a `View`-menu checkbox item) toggles `pip_auto_summon`; the change persists across launches (atomic read-modify-write, like `pip_layout`).
- [ ] The PiP on/off **manual** state persists per the bundle-identity isolation alongside `pip_layout`, **OR** defaults off-on-launch — decide + document (leaning: persist last manual state; auto-summon then layers on top). *(Carried as Open Question Q2.)*

## Out of Scope

- **PiP click-to-focus** — unchanged Future Possibility / Anti-Goal. The panel stays display-only; auto-summon changes *when it appears*, not *what clicking a tile does*.
- **Menu-bar "Toggle PiP" entry** — that's an M7 (menu-bar) surface. WP5 builds the in-Claudesk toggle only; note the forward-coupling.
- **AwaitingInput-triggered summon** — explicitly rejected as a trigger (it stays an in-panel emphasis signal). Reconsidering it is future work, not WP5.
- **Multi-monitor "is Claudesk visible on the other display" detection** — there is no clean macOS signal to distinguish "unfocused but visible on another monitor" from "unfocused and hidden"; we do NOT attempt it. The off-switchable setting is the mitigation (documented known limitation).
- **Sticky / timeout / never auto-dismiss variants** — dismiss is immediate-on-refocus only.

## Technical Constraints

- **Lead technical risk (resolve at plan, likely a quick probe): reliable main-window blur/focus detection.** The summon trigger depends on Claudesk knowing when its *main window* loses/gains focus **system-wide** (⌘Tab to another app, Mission-Control Space switch, minimize) — not just intra-Claudesk focus changes. Tauri exposes `WindowEvent::Focused(bool)`. The known wrinkle: the PiP is a **`NonactivatingPanel`** (`can_become_key_window:false`, `hides_on_deactivate:false`) — showing/hiding it must NOT itself trigger a main-window focus event that re-fires the state machine (a summon→focus-event→dismiss→focus-event loop). The non-activating design *should* prevent this (the panel never takes key/main status), but it must be verified. Also verify the debounce-cancel correctly fires on refocus regardless of *how* the user returns (⌘Tab vs Mission Control vs Dock click — all should be a `Focused(true)`).
- **Settings store seam (WP4, as-built).** `config_store/settings.rs` holds `AppSettings { pip_layout: Option<PipLayout> }` in `settings.json` (app-data dir, bundle-identity-isolated). It is reachable from BOTH the main webview AND the PiP webview via IPC (the PiP is a separate webview heap — localStorage would NOT be shared). Add `pip_auto_summon: Option<bool>` as a sibling optional field; add `read_pip_auto_summon` / `write_pip_auto_summon` mirroring the existing `read_pip_layout` / `write_pip_layout`.
- **PiP toggle/teardown seam (WP1/WP3/WP4, as-built — `src-tauri/src/pip/commands.rs`).** `pip_toggle(app) -> Result<bool>` builds-once then shows/hides; `teardown(app)` closes via `to_window()`→`close()` on main-window close. Two broadcast events: `PIP_VISIBILITY_EVENT = "pip-visibility"` (bool — the main-webview mirror-cost gate listens) and `PIP_LAYOUT_EVENT = "pip-layout"`. Panel label = `"pip"`. **The borderless `NonactivatingPanel` is load-bearing** — dropping `.borderless()` re-crashes (WP1); closing a live panel directly is a UAF abort (dismiss via toggle, tear down via `to_window().close()`). Auto-summon/dismiss must drive the **same** show/hide path as the manual toggle (not a second code path), so the visibility broadcast + cost gate stay coherent.
- **Status broadcaster (M3, as-built).** Roster + status come from `status_broadcaster` (`workspace-status` event, cwd→workspace registry). Workspace open→`workspace_register` / close→`workspace_deregister`. The PiP subscribes to the same `workspace-status` event the filmstrip does (snake_case `WorkspaceStatusUpdate` DTO). The roster-tracking AC reuses this — no new registry.
- **`app_menu` seam (as-built, native menu bar SHIPPED 2026-06-24, commit `f815154`).** The `View`-menu toggle reuses the existing id→event→action menu seam (no accelerator).
- **No 3rd-party API / SDK dependency.** Pure in-app (AppKit focus events via Tauri + local settings file). No probe-WP gating from an external service. The only "unknown" is the in-house focus-event behavior above — a code probe at plan time, not a 3rd-party integration probe.
- **MCP `tauri` bridge verify posture.** Per the WP2 verdict + CLAUDE.md: the agent can drive live verify-self for the main webview AND the PiP panel (`windowId:'pip'`) via `mcp__tauri__*`. Focus-event behavior (blur/refocus across apps/Spaces) and the installed-`.app` NSPanel parity are operator-only at verify-human (the webview can't see real cross-app focus; the bridge attaches to the running app but synthetic app-switching is not high-fidelity). Carry those to WP6 / verify-human.

## Open Questions

Leaning: **stay up** (a manual summon is a deliberate "I want this visible" act; only auto-summoned panels auto-dismiss). Decide + document at plan.
- [ ] **Q3 — exact debounce value + whether it's user-tunable.** Spec says ~3s. Confirm a sensible default (3s? 5s?) and whether to expose it (leaning: hard-coded constant in v1, not a setting — one less knob).
- [ ] **Q4 — focus-detection probe outcome (the lead technical risk above).** Confirm `WindowEvent::Focused` fires reliably for cross-app / Space-switch blur, and that showing/hiding the non-activating panel does not re-trigger it. If it *does* re-trigger, the plan needs a guard (e.g. suppress focus-event handling for a tick around a programmatic show/hide).
