# Feature: Dev/Prod isolation (concurrent installed-app + `pnpm tauri dev`)

**Workflow:** feature

## Problem Statement

To dogfood Claudesk (use the installed `.app` to develop Claudesk itself), the operator must run the installed prod app and `pnpm tauri dev` **at the same time** — the dev instance is how they verify changes while the prod instance stays their daily driver. Today the two builds are not isolated: both resolve to the bundle identifier `com.claudesk.app`, so they share one `app_data_dir()` (→ same `projects.json`, same `hook.sock`, same deployed `claudesk-hook.pl`) and register an **identical** hook command in the global `~/.claude/settings.json`. Concurrently, the second instance to launch unconditionally `remove_file()`s and re-binds the shared socket (`hook_socket/mod.rs:109-116`), silently stealing the CC status stream from the first; and because the settings.json detection marker is the bare basename `claudesk-hook.pl`, the two builds would fight over a single global registration. All five collisions are downstream of one root cause — the shared identifier. **Fix (Option A, operator-decided): make the running app's bundle identifier the single source of truth, and derive the hook script basename + socket + data dir + registration marker from it at runtime.** A `tauri.dev.json` config overlay gives the dev build identifier `com.claudesk.app.dev`; everything else isolates automatically or via an identifier-derived basename. The Perl hook is *already* multi-instance-safe (best-effort connect, 1s timeout, silent no-op when its socket isn't bound — verified by reading the live deployed script), so event fan-out to both sockets is free. This BLOCKS M4 finalize: M4's dogfood-replace point is unreachable until concurrent dev+prod works without cross-talk.
