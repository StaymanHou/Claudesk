## Problem Statement
Claudesk spawns `claude` with `--dangerously-skip-permissions` ("yolo") **unconditionally** (`cc_session/mod.rs:43` const, pushed into argv at `mod.rs:232`). A friend-user wants to be able to open CC **without** yolo so CC shows its permission prompts. Yolo stays the vision-explicit default ‚Äî this is the long-anticipated "Phase 4 setting will let the user opt out," landing early as an **opt-out only** toggle. Gate the yolo argv flag on a new app-global `AppSettings.cc_yolo: Option<bool>` field, default-when-unset = `true` (yolo ON). Because the flag is chosen once per CC process, a toggle takes effect on the **next** CC spawn, not the live session.

## Plan-time decisions (recorded)
- **App-global, not per-project.** `cc_yolo` lives in `AppSettings` (`config_store/settings.rs`), mirroring `pip_mode`/`pip_layout` ‚Äî bundle-identity-isolated, atomic read-modify-write. The WBS leaned app-global ("an opt-out toggle"); a per-project variant would be a `Project` field in `projects.json`, which is heavier and not what the friend asked for. **Decided: app-global.** `[PRIOR: operator-helpful-friend-misfiring-as-offswitchable-setting]` agrees (Rule 2) ‚Äî ship off-switchable, default to operator benefit (yolo ON).
- **Affordance = a single `CheckMenuItem` in the View menu.** No global Settings UI exists yet (that's M12). The established minimal-affordance pattern is the native menu: the PiP tri-state radio (`PIP_MODE_*` CheckMenuItems ‚Üí `FUNCTIONAL_IDS` ‚Üí `menu` event ‚Üí `App.tsx` listener ‚Üí `invoke`) is the exact template. A boolean yolo toggle is a strictly simpler instance (one CheckMenuItem, checked = yolo ON). **Decided: one View-menu `CheckMenuItem`** "Skip Permission Prompts (yolo)", checked when on. No accelerator (matches the native-menu no-new-hotkeys pattern).
- **Read at spawn time** (`cc_spawn` already has `AppHandle` ‚Üí `app_data_dir()` ‚Üí `read_cc_yolo`). Documented as next-spawn, not live-toggle.
