## SURFACE-2026-07-13-M9-WP4-MINUTE-QUANTIZATION-ZEROES-SUBMINUTE-AI-DOING
- **Source:** feature:verify-human (M9 WP6b-2 Phase 1 ‚Äî operator noticed neo-stayman-assistant's afternoon usage under-reported in the Week view). ROOT CAUSE traced during the same investigation.
- **Target level:** feature (WP4 `time_store::query` duration-summing bug) ‚Äî NOT a WP6b-2 defect (render is faithful); also NOT the WP3 reclassifier classification logic (that's correct ‚Äî it builds the AiDoing segments fine).
- **Symptom:** Dense back-to-back tool-use sessions report `ai_doing: 0`. neo `df2a3051` (92 tool events, 11 min) ‚Üí `{ai_doing:0, reviewing:11}`; `ea577ad8` (16 min) ‚Üí `{ai_doing:0, reviewing:15, subagent:1}`. The Week "total" pill sums only the AI family (`cellActive`), so neo's real ~27 painted minutes show as a "1m" pill ‚Üí "why isn't my afternoon work showing up?"
- **ROOT CAUSE (confirmed by code read + live data):** a **minute-quantization** bug in `src-tauri/src/time_store/query.rs`. The reclassifier `ai_segments_for_window` correctly builds `AiDoing` segments from the Pre‚ÜíPost `tool_intervals` (verified: neo's events all carry matching `tool_use_id`s; intervals pair fine). BUT the query layer converts each segment's ms endpoints to **integer minutes** via `ts_to_minutes = (ts_ms - day_start_ms).div_euclid(60_000)` (L181), THEN sums per-kind minutes as `seg.end - seg.start` on the FLOORED values (day L363, week L709). AI tool-execution is intrinsically **sub-minute** (Pre‚ÜíPost ~1s apart, e.g. 18:48:49‚Üí18:48:50) ‚Üí both endpoints floor to the SAME integer minute ‚Üí `end - start == 0` ‚Üí the segment contributes **0** to `ai_doing`. The between-tool `reviewing` gaps are minute-scale so they survive ‚Äî hence `ai_doing:0` + inflated `reviewing`. Reference `reclassify.py` never hit this: it summed durations in **ms** and used the minute coordinate only for render positions; the Claudesk port sums AFTER quantizing.
- **Context:** verified via dev SQLite (`com.claudesk.app.dev/time-analytics.sqlite`) ‚Äî raw events present + correctly date-bucketed (WP2 capture + WP4 bucketing fine). This corrupts the milestone's core "measure, don't infer" value prop + every AI-vs-human duration number (sub-minute AI work vanishes; human-review time over-reported). Frozen-contract note: WP1 mandates `SegPayload.start/end` as integer minutes-from-midnight, so the fix is NOT "make the contract sub-minute" ‚Äî it is to **compute per-kind duration totals at ms (or finer) precision BEFORE applying the minute coordinate**, so a sub-minute segment still contributes its real duration to the rollup even though its rendered start/end round to the same minute.
- **Suggested action:** `/feature-reproduce` first ‚Äî a query/reclassifier unit test with N sub-minute Pre‚ÜíPost tool pairs asserting `ai_doing > 0` (currently 0). Deterministic repro: the neo sessions are in the dev DB. Fix in `query.rs` duration-summing: preserve ms/seconds precision when accumulating per-kind totals; keep the integer-minute `start`/`end` on `SegPayload` for rendering (contract-preserving). Re-verify neo shows realistic `ai_doing` day + week.
- **Priority:** high (wrong core numbers; the milestone's reason-for-being is "measure, don't infer" ‚Äî this silently zeroes the thing being measured. NOT blocking WP6b-2 render phases, but must be fixed within M9 before WP7 deprecates the standalone tool + exit-verifies)
- **Status:** RESOLVED 2026-07-13 (feature `m9-fix-minute-quantization-ai-doing`, local/uncommitted per commit-only-when-asked). Fix: `dur_ms` on `SegPayload` + sum-on-`dur_ms` + `ms_to_minutes_round` (round-half-up, once per per-kind total); `start/end` stay minute-quantized (contract intact). 3 regression pins; live-verified against the real dev DB (neo `df2a3051` ai-doing 0 ‚Üí ~57s). Reproduced red-green via `/feature-reproduce` as suggested. Archived at `workflow/archive/m9-fix-minute-quantization-ai-doing.md`. ## SURFACE-2026-07-08-M9-SESSION-TERMINATION-NOT-TRACKED
- **Source:** feature:verify-human (M9 WP6a Phase 3 ‚Äî operator spotted at re-confirm)
- **Target level:** feature (NEW M9 feature ‚Äî its own `/feature-spec`; NOT a WP6a leaf. Operator decision 2026-07-08: FIX NOW inside M9, do not defer.)
- **Type:** gap (data-model / session-lifecycle semantics)
- **Summary:** The tracker has **no concept of session termination**. A session's `end` is just the timestamp of its last hook event; a dangling (dead) session is indistinguishable from a long idle pause, so a closed/crashed session's row stretches on and its trailing gap is classified `away` ‚Äî making the dashboard show a session as effectively "running all day" when no such session is alive (operator saw scratch-b 10:54‚Üí13:42 with no live scratch-b workspace). Root: CC hooks fire on activity; there is no observed teardown, so app-kill / CC-crash / power-outage all look identical to "idle".
- **Context:** Upstream of WP6a ‚Äî lives in the WP2 hook writer (`src-tauri/src/time_store/`) + WP3 reclassifier (how `end` + trailing state are derived). The dashboard only *renders* what the reclassifier produces; the fix is NOT in the dashboard. Affects correctness of every duration/away/longest-session stat.
- **Operator-chosen design direction (2026-07-08, all four signals ‚Äî record as intent, build in the spec):**
  1. **Explicit end on workspace close** ‚Äî Claudesk knows when it kills a PTY / closes a workspace ‚Üí write an explicit session-end marker event (a clean close is recorded exactly). Likely a new event `source`/`event` kind on the existing gated `HookEvent`/time-store drain.
  2. **Max-idle cap** ‚Äî after N minutes with no events, cap a dangling session's `end` (so crash/outage sessions don't stretch to the day-window edge). Reclassifier-side.
  3. **CC `SessionEnd` hook IF it exists** ‚Äî register it as the authoritative end signal. **MUST verify via a live hook-stream capture first** (per `cc-hook-capture-beats-docs` / `cc-hook-event-facts` memories ‚Äî docs can be wrong about event existence). If it exists, it's the cleanest signal; if not, rely on 1/2/4.
  4. **Startup reconciliation** ‚Äî on app launch, close out sessions left dangling by a prior crash/outage (`end` = last-seen event), so the power-outage case self-heals.
- **Interactions to resolve in the spec:** precedence when signals disagree (explicit-end vs. max-idle-cap vs. a late event after a presumed end); what the max-idle N should be vs. the reclassifier's existing away-threshold (avoid double-defining "idle"); whether the explicit-end marker is a new `event` value on the current schema or a new column; migration of already-dangling historical rows.
- **Suggested action:** open as a NEW M9 feature via `/feature-spec` after WP6a closes (WP6a Phase 4 lazy-load still finishes first). Spec must FUSE the four signals into one coherent session-end model + verify the CC `SessionEnd` hook empirically before depending on it.
- **Priority:** high (operator: fix now inside M9; makes the dashboard's core numbers trustworthy)
- **Status:** RESOLVED 2026-07-08 (M9 WP6.5 shipped ‚Äî `workflow/archive/m9-wp6.5-session-termination-model.md`). All four signals built: explicit `NativeSignal::WorkspaceClose` marker (cc_kill/kill_all) + 30-min max-idle cap (idle-gap keyed, `SESSION_IDLE_CAP_MS`) + CC `SessionEnd` honored (empirically confirmed to fire on clean close, `reason`‚Üímeta) + startup `reconcile_dangling`. D3 precedence explicit>SessionEnd>cap>live; read-time capping self-fixes historical rows (no destructive backfill). Reconciliation LIVE-proven via MCP bridge (seed‚Üírelaunch‚Üíclose at true ts; idempotent). 498 lib + 6 integ tests, clippy --all-targets clean. Dashboard-visual "no longer running all day" carried to release gate. Files: `src/App.css` (`.filmstrip-tile-mirror` + header row), `src/components/workspace/XtermPane.tsx` (the serialize call), possibly `src/components/workspace/Filmstrip.tsx` (tile layout) + `terminalMirror.ts`/`mirrorTicker.ts`.
