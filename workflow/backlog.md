## SURFACE-2026-06-26-ABSORB-CLAUDE-TIME-INTO-CLAUDESK
- **Source:** operator request + analysis (2026-06-26 session) — "move the claude-time feature into this claudesk app"
- **Target level:** product (a roadmap milestone) — new universal feature
- **Type:** new-work (absorb an external tool) + tool deprecation
- **Summary:** Absorb the `claude-time` time-tracking tool (currently at `_ref/claude-customization/tools/claude-time/`) into Claudesk as a **native time-analytics panel**, and **deprecate the standalone tool** once absorbed. claude-time logs CC lifecycle hook events to a local SQLite DB (`~/.claude-time/events.sqlite`, privacy-strict: prompt *lengths* only, never content), runs a reclassifier (gap → tool/active/reading/thinking/away buckets with typing-debit + cross-session reattribution), and emits a Gantt-style HTML dashboard via a `claude-time visualize` CLI. Inside Claudesk it becomes a right-panel tab (alongside Editor/Diff/Terminal/Docs) rendering the dashboard natively — no CDN React, no separate browser window, no stale-tab cursor.
- **Why it fits Claudesk:** It serves the same "attention is the scarce resource across N parallel projects" thesis (see memory [[claudesk-philosophy]]) — the *retrospective* face of what the filmstrip/menu-bar do in *real time*. The event substrate already overlaps: Claudesk's `claudesk-hook.pl` + `hook_socket` listener + `hook_install` registration are siblings of claude-time's `hook.pl`, both Perl, both env-gated. Claudesk's cwd→workspace registry makes per-project attribution native instead of git-root-guessed. And the dashboard *wants* a webview, which is exactly what Claudesk is.
- **DECISIONS LOCKED (operator, 2026-06-26):**
  1. **Scope = full absorption, NOT reader-only.** The earlier reader-only recommendation was predicated on preserving bare-terminal / standalone-CLI coverage and avoiding cross-repo schema coupling. Operator has decided friends won't use the CLI and neither will he going forward, so **claude-time is deprecated once absorbed** — both those constraints vanish, and the "don't maintain two copies" goal makes absorption the right call. Claudesk becomes the sole owner.
  2. **It is a UNIVERSAL feature, NOT workflow-coupled.** It observes generic CC lifecycle events, not the customization *skills* — so it does NOT belong in the off-by-default "workflow-coupled" feature bucket (which is for the Docs viewer / auto-resume / skill-orchestration, all of which read skills / WIP files / `.session.md`). It lives behind its own simple Settings toggle, **default OFF** (to honor the cost constraint for friends), but a friend *can* turn it on and get value without running the workflow.
  3. **Write only when tracking is enabled.** The hook fires regardless (it drives the universal live status dots — that must stay). But the time-row SQLite write happens **only when the tracking toggle is on.** When off: receive event → update status → DO NOT touch SQLite. Cost-when-off is then literally zero storage/IO — the friend constraint is satisfied. This is *strictly cheaper* than reader-only-on-a-separate-hook (which kept a second hook registered forever) AND than naive absorption (which would write for everyone).
  4. **DB does NOT survive the dev/prod split — and SHOULD NOT.** Dev and prod each get their OWN time-tracking DB, consistent with the existing `com.claudesk.app` vs `com.claudesk.app.dev` identity isolation. Likely move the DB under `app_data_dir()` (per-identity) rather than `~/.claude-time/`. No migration of the existing `~/.claude-time/events.sqlite` history required (start fresh per identity).
- **Absorption plan (the four layers move differently — settle at `/product-wbs`):**
  - **Hook writer** → fold persistence into Claudesk's **Rust `hook_socket` listener**, NOT the Perl hook. The Perl hook already sends every field over the socket; the listener writes the time-row (gated by decision 3). Remove the separate `claude-time-hook.pl` registration from `~/.claude/settings.json`. NOTE: claude-time logs 11 event types (incl. `PreToolUse`/`PostToolUse`/`SubagentStart`/`SubagentStop`/`SessionStart`/`SessionEnd`) — MORE than the 4 Claudesk currently registers (`UserPromptSubmit`/`Stop`/`Notification` + the QoL-WP2 `PostToolUse`). Absorption must extend `claudesk-hook.pl`'s registration to the full set the reclassifier needs, and the Perl hook must forward the extra fields (`tool_use_id`, `agent_type`, `prompt_length_chars`, `source`).
  - **DB / schema** → Claudesk owns it; per-identity `app_data_dir()` location (decision 4). Schema is claude-time's `events(ts, session_id, cwd, event, tool_name, agent_type, meta)`.
  - **Reclassifier** → **port `reclassify.py` (~370 lines, pure stdlib) → Rust.** The one piece of real logic; the existing 29 assertions (`test_reclassify.py`) are the porting oracle. Keep the grouping logic here (git-root + `project_names`-style config) so behavior is one source of truth.
  - **Dashboard** → port `viz/dashboard.jsx` → a React right-panel tab. Drop the unpkg/Babel/template-inlining machinery (`viz_render.py` largely evaporates — it exists to emit a standalone file we no longer need). `viz_data.py` (events → segment-model JSON) becomes a Rust query layer.
  - **CLI (`claude-time report` / `visualize`)** → deprecated, does not move. (A text-report view could be a trivial second tab later.)
- **Open sub-decisions for `/product-wbs`:** exact DB path under `app_data_dir()`; whether to keep claude-time's `config.json` tuning surface (`chars_per_sec`, thresholds, `project_names`) as a Claudesk setting or hardcode; how the panel scopes (global all-projects view vs. per-workspace) given it's a per-workspace tab; the Settings-toggle plumbing (this is also the first concrete instance of the universal-vs-workflow-coupled feature-flag split — worth designing the toggle pattern here even though this feature isn't in the workflow-coupled bucket).

## SURFACE-2026-06-25-FILMSTRIP-MIRROR-BANNER-OCCLUDED-AT-SESSION-START
- **Source:** operator observation (2026-06-25, during the M5 WP1 session)
- **Target level:** feature (small) / bug-shape — filmstrip live-mirror anchoring
- **Type:** bug-shape (early-session content occluded by the tile header)
- **Summary:** In a filmstrip tile's live mirror (the ~1 fps `serializeAsHTML()` render of a background workspace's CC terminal), the **CC banner + first ~10–20 lines of a fresh session are occluded by the tile's project-name + status-dot header row** (top of the tile). Once output grows past the viewport, tailing works fine — but the early short-content phase looks broken. **Desired behavior:** the mirror should "fill from the bottom first" — short content bottom-anchors (sits at the tile's bottom edge and grows UPWARD, clear of the top header), then tails normally once it overflows the viewport (operator-confirmed reading 2026-06-25).
- **Root cause (analyzed 2026-06-25, NOT yet fixed):** The mirror block is already bottom-anchored at the *tile* level (`.filmstrip-tile-mirror { position:absolute; bottom:0; transform-origin:bottom-left }` in `src/App.css` ~L204) and tails via `serializeAsHTML({ scrollback: 40 })` (`src/components/workspace/XtermPane.tsx` ~L211–225). BUT the serialized block is `active-screen + 40-row scrollback`, and the active screen is the FULL terminal height (~40 rows) even when only ~10–20 rows have real content. So at session start the banner content sits near the TOP of a tall, mostly-blank block; bottom-anchoring the *block* pushes that top content up to the tile's top edge — exactly under the header row. The existing bottom-anchor handles "tall block tails to newest"; it does NOT handle "sparse block → real content should sit at the bottom."
